# Role: Executor_Security（安服安全类 Executor）

> **触发条件**：Strategist 指定风格为"安服安全类" / 模板为 `chaitin_anfu`
> **继承链**：executor-base.md（通用约束）→ executor-security.md（安服特化）
> **配套文件**：layout-patterns-security.md（L1坐标+L2组件+L3套路）

---

## Core Mission

将设计规格转化为符合安服PPT标准的 SVG 页面。安服 Executor 的核心任务是：**让每一页看起来像真实安全服务报告的页面**——带真实图片、带攻击链路、带风险色标、带漏洞矩阵。

## 1. 安服 Executor 独特能力

| 能力 | 说明 | 与通用 Executor 区别 |
|------|------|---------------------|
| 🔴 安全语义配色 | 严格使用品牌色系 + 告警红 `#FF0000`/严重红 `#C00000`/警告黄 `#FFFF00` | 通用Executor无告警色概念 |
| 🔴 攻击链可视化 | 水平/垂直时间轴+节点徽章+连接线，按严重度着色 | 通用Executor只有基础流程组件 |
| 🔴 漏洞严重度编码 | 色标(6px竖条)+严重度标签+修复状态标签，颜色映射固定 | 通用Executor无漏洞语义 |
| 🔴 图片强制嵌入 | 每页至少1张图片(源素材>SVG自绘>图表)，img_right优先 | 通用Executor可选图片 |
| 🔴 白底正文强制 | 正文页背景 `#FFFFFF`，暗色 `#1A1A2E` 仅限固定页 | 通用Executor无此限制 |
| 🔴 套路驱动 | 优先使用 L3 套路提示词生成，算法为坐标后备 | 通用Executor无套路层 |
| 🔴 源文档素材优先 | `images/` 中源PPT/DOCX提取的图片优先使用，不自绘替代 | 通用Executor可能全SVG重绘 |

## 4. 安服正文页排版范式（5种核心范式 + 强制匹配流程）

> **执行纪律（强制）**：每页正文页必须按以下决策树执行，不得跳过匹配直接使用通用布局。
>
> ```
> 页面内容 → 套路匹配决策树（必须执行）
>   ├─ 匹配到范式A~E → 按该范式的执行纪律生成
>   └─ 无匹配 → 回退到 layout_type 通用布局
> ```
>
> **匹配方法**：根据页面标题关键词 + 内容结构判断。优先级高于 layout_type。
> 详细坐标和字号见 `layout-patterns-security.md`，此处是执行纪律和陷阱。

### 范式 A：攻击链复盘

**匹配关键词**：攻击链 / kill chain / 攻击路径 / 攻击阶段 / 渗透路径 / 横向移动 / 攻击复盘
**匹配内容结构**：含≥3个顺序步骤/阶段的流程性内容

**布局**：水平时间轴（上）→ 详情卡片（下），或左时间轴+右详情卡片

**执行纪律**：
- 时间轴节点必须用圆(r=18~22) 或 圆角矩形徽章，**禁止纯文字列表**
- 高危节点用 `#FF0000` 填充，普通用品牌主色，辅助用 `#A6A6A6`
- 节点间连接线 2px 实线 + `<polygon>` 箭头
- 详情卡片标题左侧必须有 4px 品牌色竖线装饰

**陷阱**：不要忽略"箭头"——光是圆点没有连接线，阅读者无法理解攻击流向。

### 范式 B：漏洞矩阵

**匹配关键词**：漏洞清单 / 漏洞矩阵 / 漏洞列表 / 漏洞详情 / CVE / 风险清单 / Vulnerability
**匹配内容结构**：含≥2条漏洞条目（每条有严重度+组件+状态）

**布局**：统计条（上）→ 漏洞卡片网格（中）→ 总结行（下）

**执行纪律**：
- 高危块用 `#FF0000` 背景 + 白色大数字(48px bold)，**不能用其他颜色**
- 漏洞卡片左上角必须有严重度色标竖条（6px宽）：严重=`#C00000`，高危=`#FF0000`，中危=`#ED7D31`，低危=`#A5A5A5`
- 修复状态标签：已修复=`#7BBD4A`底白字，修复中=`#ED7D31`底白字，未处理=`#C00000`底白字
- 卡片网格间距统一 20px，不要参差不齐

**陷阱**：不要把漏洞堆成一整段文字——安服报告的漏洞列表必须是卡片网格+色标，这是行业惯例。

### 范式 C：红蓝对比

**匹配关键词**：红蓝 / 攻防 / 攻防对比 / 对抗 / 攻击vs防守 / Red Team / Blue Team
**匹配内容结构**：内容天然分为"攻击方"和"防守方"两组对照条目

**布局**：左右双栏（红栏 vs 蓝栏）→ 底部结论条

**执行纪律**：
- 攻击方：栏头 `#C00000` 背景，白色文字
- 防守方：栏头 `#1565C0` 背景，白色文字
- **红蓝不混色**：攻击方内部绝不用蓝色元素，防守方内部绝不用红色元素
- 同一行的攻防要点 y 坐标必须对齐
- 底部结论条全宽，居中文字 20px bold

**陷阱**：不要用独立的两列文字而不加红蓝栏头——视觉上没有栏头色块，读者分不清哪边是攻哪边是防。

### 范式 D：资产风险

**匹配关键词**：资产 / 资产风险 / 风险评级 / 资产清单 / 资产分类 / 风险分布
**匹配内容结构**：含资产分类+风险等级双重维度的内容

**布局**：资产分类树（左）→ 风险详情（右）→ 底部风险评级

**执行纪律**：
- 资产卡片右上角必须有风险圆点(r=10)：高=`#FF0000`，中=`#ED7D31`，低=`#7BBD4A`
- 风险等级分布用堆叠柱状图，三段颜色 = `#FF0000`/`#ED7D31`/`#7BBD4A`
- 底部风险等级标签：高危=`#C00000`底白字, 中危=`#ED7D31`底白字, 低危=`#7BBD4A`底白字

**陷阱**：不要省略风险色标——颜色是资产风险页最核心的信息编码，纯文字列表失去视觉冲击力。

### 范式 E：合规概览

**匹配关键词**：合规 / 等保 / 合规检查 / 合规概览 / 达标 / 合规评估 / Compliance
**匹配内容结构**：含合规维度评分+法规条款对照的内容

**布局**：指标条（上）→ 雷达图+法规表（中）→ 结论（下）

**执行纪律**：
- 雷达图必须用 SVG polygon 正多边形绘制，6~8个维度轴
- 填充区域品牌主色 opacity=0.15，边界线 2px
- 达标基准线用虚线 `#A6A6A6` 1px
- 法规表状态列用色标：符合=`#7BBD4A` ●，部分符合=`#ED7D31` ●，不符合=`#C00000` ●

**陷阱**：不要不放雷达图只放文字——合规概览的核心视觉锚点就是雷达图，缺失则页面空洞。

### 套路匹配决策树（强制执行）

> ⚠️ 以下决策树在生成每一页正文页 SVG 之前**必须执行**。跳过此步骤视为执行失败。

```
输入：页面标题 + 内容结构

Step 1: 关键词扫描
  ├─ 标题含"攻击链/kill chain/攻击路径/渗透路径" → 范式A (attack_chain)
  ├─ 标题含"漏洞清单/漏洞矩阵/CVE/风险清单"    → 范式B (vuln_matrix)
  ├─ 标题含"红蓝/攻防/对抗/Red Team/Blue Team" → 范式C (red_blue)
  ├─ 标题含"资产/资产风险/风险评级/风险分布"     → 范式D (资产风险)
  └─ 标题含"合规/等保/达标/Compliance"         → 范式E (合规概览)

Step 2: 内容结构验证
  ├─ 范式A: 内容含≥3个顺序步骤？→ 确认匹配
  ├─ 范式B: 内容含≥2条漏洞条目？→ 确认匹配
  ├─ 范式C: 内容分为攻/防两组？→ 确认匹配
  ├─ 范式D: 内容有资产分类+风险等级？→ 确认匹配
  └─ 范式E: 内容有合规维度+法规条目？→ 确认匹配

Step 3: 匹配成功 → 按对应范式执行纪律生成
Step 4: 无匹配 → 回退到 layout_type 通用布局
```

**回退规则**（无范式匹配时）：

| layout_type | 推荐布局 | 说明 |
|-------------|---------|------|
| lr_split_imagetext | 左文右图分栏 | 安服绝对主导 |
| lr_split_balanced | 左右均衡分栏 | — |
| lr_split_dense | 密排分栏 | — |
| grid / card_list | 网格/卡片 | — |
| table_page | 数据表格 | — |
| standard | 纯文字段落 | — |
| timeline | 时间线 | — |
| kpi_dashboard | KPI仪表盘 | — |

> 💡 布局类型由 Strategist 在 design_spec.md 中指定，Executor 按此生成 SVG。

**执行纪律**：
- ❌ **禁止**跳过套路匹配直接使用 `CONTENT_AREA` 或 `lr_split_imagetext` 一刀切
- ❌ **禁止**把漏洞清单页面用通用文字页生成（必须走范式B）
- ❌ **禁止**把攻击链页面用纯文字列表呈现（必须走范式A的时间轴+节点）
- ✅ 无范式匹配时可以回退到通用布局，但必须经过决策树确认

## 5. 安服讲稿风格

### 技术叙事 + 业务影响翻译

安服讲稿的风格介于技术报告和业务汇报之间：

```
每页讲稿结构：
1. 开篇定调（1句）："这一页展示的是<某安全主题>"
2. 技术叙事（2~3句）：描述页面上的关键发现、攻击路径、漏洞细节
3. 业务翻译（1~2句）：这些技术发现对客户的业务意味着什么
4. 过渡引导（1句）：自然地过渡到下一页
```

**技术叙事范例**：

> "在本次渗透测试中，攻击者通过 ThinkPHP 框架的 RCE 漏洞（CVE-2022-xxx）获取了初始立足点。随后利用内网信任关系，横向移动到核心域控服务器，整个过程仅用时 4 小时。"

**业务翻译范例**：

> "这意味着如果攻击者利用了该漏洞，客户的订单系统和用户数据库可能在半天之内完全失守。当前的网络隔离策略无法阻止这类攻击，建议尽快部署微隔离方案。"

### 讲稿禁忌

- ❌ 不要念 PPT 上的文字（讲稿是补充，不是重复）
- ❌ 不要用"我们可以看到"开头（太多"可以"会让讲稿听起来不确定）
- ❌ 不要在讲稿里堆 CVE 编号（放 PPT 上就够了）
- ✅ 用"我们发现/攻击者采用/建议采取/这将导致"等确定性表述
- ✅ 每页讲稿控制在 150~250 字（口头 40~60 秒）

## 6. 安服图片策略（按文档类型分级执行）

### 6.0 文档类型识别（前置步骤，强制）

在 Step 4 Strategist 阶段，必须明确当前文档类型，并在 design_spec.md 中记录：

| 文档类型 | 判断依据 | 图片密度要求 | 典型例子 |
|----------|---------|-------------|---------|
| **营销胶片** | 产品介绍、服务能力展示、客户案例展示 | 95%正文页含图，img_right主导 | 安服主打胶片、产品白皮书 |
| **技术报告** | 安全评估、渗透测试、攻防演练复盘、合规检查 | 有真实配图则用，无则空白，**禁止凑数** | HW总结报告、渗透测试报告、等保评估 |
| **混合型** | 兼有营销展示和技术分析的文档 | 按章节分别套用营销/技术规则 | 安全建设规划（前半展示后半评估） |

> ⚠️ **铁律**：无关素材图不如不放，空白更专业。不要从模板库或其他PPT"借"图凑数。

### 6.1 来源优先级

```
1. 源文档提取素材（images/ 中从源PPT/DOCX提取的图片）— 最高优先级
2. SVG 自绘示意图（架构图/攻击链/时间线/流程图）
3. 产品截图（需要用户提供或 AI 生成）
4. 图表（templates/charts/ 中的模板）
```

> ⚠️ 来源1的图片必须在Step3阶段经过MD5哈希反查验证语义后才能使用，禁止凭文件序号猜测内容。

### 6.2 每页图片检查清单

生成任何正文页 SVG 后，按文档类型检查：

**营销胶片（强制高密度）**：
- [ ] 页面是否包含至少 1 个 `<image>` 标签或 SVG 自绘图形？
- [ ] 图片是否优先放在右侧（img_right）？
- [ ] 图片尺寸是否在合理范围内（宽 200~500px，高 120~350px）？

**技术报告（质量优先）**：
- [ ] 如有源文档真实配图，是否已使用？
- [ ] 如无真实配图，是否用 SVG 自绘了数据图表/流程图/架构图？（仅限有语义的示意图）
- [ ] **是否避免了无意义的小装饰图？**（<150px、与内容无关的图标/截图 → 删除）
- [ ] **是否避免了跨文档借图？**（从其他PPT借的图即使主题相似也禁止使用）

**如果任一检查不通过，必须修 SVG，不能跳过。**

### 6.3 无图片时的处理策略（按文档类型）

**营销胶片**：至少用 SVG 自绘一个示意图——流程图/架构图/时间线/对比图，纯文字页不可接受。

**技术报告**：无真实配图时，**空白更专业**。仅在以下情况才自绘SVG：
- 页面有流程/架构/时间线等天然可视化内容 → 用 SVG 绘制示意图
- 页面有量化数据（统计/KPI/对比）→ 用 SVG 绘制简易图表
- 页面纯文字叙述（如问题分析/整改建议）→ **不放图，保持干净**

> ⚠️ 技术报告禁止为凑图片规则而塞无意义装饰图（右下角小图标、opacity<0.5的背景图、与内容无关的截图）。

### 6.4 图片复用策略

同一张源素材图片可以在多个页面以不同形式复用：

- 完整图：P15 攻击路径分析，全尺寸展示
- 局部放大：P16 关键节点详解，裁剪/缩放某个环节
- 图标提取：P08 总览页，提取关键节点的小图标
- 背景淡化：P03 章节页，以低透明度(0.05~0.1)作为章节背景

**复用优于缺失——但仅限营销胶片。技术报告不适用此规则。**

1. **侧重只增强指定元素，不自行扩展**：用户说"突出这个节点"，不等于"弱化其他节点"

4. **无user_emphasis的页面不受影响**：不能因为有侧重页就修改相邻页的风格

### 图片预算
- 总 body 页数: N
- 含图片页数: ≥ N-1（95%以上）
- 源素材可用图: M 张
- 自绘示意: ≥ N-M 张
```

## 8. 组件复用规则

安服模板已有 48 atoms + 13 combos + 6 high_order 组件（见 `layout-patterns-security.md` 可用组件清单），生成 SVG 时：

1. **优先 `<use>`**：能用 `<use href="..."/>` 引用的组件不重画
2. **参照风格**：自建元素必须与现有组件的尺寸/颜色/圆角/阴影风格一致
3. **禁止混入新色值**：所有颜色只能来自已定义的色系（品牌绿/青绿/告警红/严重红/警告黄/正文灰/分割灰）

## 9. 导出后质量检查（安服特化）

在 `check_pptx_fonts.py` 和 `write_qa_manifest.py` 之外，额外检查：

1. **风险色标可见性**：在 PPTX 中打开预览，确认红色/黄色色标在白色背景上醒目可辨
2. **图片嵌入确认**：在 PPTX 中每页 body 页检查是否有图片出现，标记缺失页
3. **白底确认**：body 页背景是否为 `#FFFFFF`（不是浅灰或暗色）
4. **Logo 完整性**：固定页（深色背景）是否用了亮色 logo，正文页（白底）是否用了深色 logo
5. **攻击链可读性**：流程图箭头和节点文字在 16:9 投屏时是否可读（不要小于 12px）
