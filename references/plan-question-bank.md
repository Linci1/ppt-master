# `/plan` 阶段标准提问清单

> 用途：当 `ppt-master` 以 Agent 模式启动、且任务属于“新建 PPT”时，优先使用本题库做需求澄清。

---

## 一、使用原则

- 目标不是多问，而是尽快形成可执行 brief
- 每组问题都应尽量帮助用户收敛，而不是把负担都丢给用户
- 如果用户信息不足，可先给出建议方向，再让用户确认
- 若用户已经明确说过的内容，不重复追问

---

## 二、七组核心问题

### 2.1 场景与行业

至少确认：

- 这份 PPT 属于什么场景：售前、汇报、总结、方案、案例证明、品牌介绍、培训、答辩
- 行业是什么
- 这个行业有没有固定表达风格、常见关注点或忌讳

### 2.2 展示对象

至少确认：

- 给谁看：技术团队、管理层、采购、客户领导、内部团队，还是混合受众
- 受众更关心：过程、结果、方案、可信度，还是 ROI

### 2.3 本次目标

至少确认：

- 希望这份 PPT 最终达成什么目标
- 看完后希望对方形成什么判断
- 看完后希望对方做什么动作

### 2.4 展示重点

至少确认：

- 本次最想突出哪 3 件事
- 更偏成果展示、能力证明、体系说明、方法论讲解、案例复盘，还是综合表达
- 哪些内容必须保留，不能被压缩掉

### 2.5 品牌与风格要求

至少确认：

- 是否必须使用某个模板
- 是否有历史案例风格要对齐
- 更偏稳妥清晰，还是高信息密度、专业感强
- 是否希望适度复杂页，还是尽量简洁

### 2.6 材料与资源

至少确认：

- 当前有哪些输入：docx / pdf / markdown / 历史 PPT / 图片 / 截图 / 图表 / 数据
- 是否允许使用文档原图
- 是否允许 AI 生图
- 是否有必须沿用的品牌素材

### 2.7 交付约束

至少确认：

- 页数范围
- 时间限制
- 是否需要偏可讲述 / 偏可阅读
- 哪些表达、术语、视觉形式必须避免

---

## 三、推荐追问方式

### 3.1 当用户目标模糊时

不要只问“你想要什么风格”，而建议问：

- 这次更偏“证明能力”“展示成果”“解释方案”还是“推动决策”？
- 如果只能让客户记住 3 件事，希望是哪 3 件？
- 希望客户看完后更相信什么？

### 3.2 当用户说“做高级一点”时

建议改问：

- 希望高级体现在：图形复杂度、内容深度、案例证明力度，还是整体专业感？
- 哪些页值得做复杂页，哪些页只需要讲清楚即可？

### 3.3 当用户给了历史案例时

建议追加确认：

- 这份案例里，你最认可的是品牌骨架、内容逻辑，还是复杂页表达？
- 是希望整体贴近，还是只借鉴其中几类页型？

### 3.4 当任务命中安服 / 长亭模板时

如果 `/plan` 已明显识别为安服或长亭品牌场景，建议不要只继续问通用问题，而要补一层行业化追问。

对安服 / `security_service`：

- 最终更要证明：能力、结果、可信度，还是整改建议？
- 哪些页值得做复杂页：攻击链、证据链、治理闭环、能力地图，还是尽量少做？
- 文档里的截图、证据图、攻击路径素材，是否允许直接进入 PPT？
- 如果受众同时包含管理层与技术团队，主副线更偏哪边？

对长亭品牌通用 / `chaitin`：

- 这次更适合 `长亭通用墨绿色` 还是 `长亭安服`？
- 长亭品牌元素要固定到什么程度：Logo、背景、页脚、保护区是否都必须保留？
- 正文是偏稳妥品牌表达，还是在骨架固定下做更丰富的内容页？

---

## 四、Plan 阶段标准输出

完成提问后，应沉淀成以下文件：

- `project_brief.md`
- 如需拆分，可再补：
  - `customer_requirements.md`
  - `content_priorities.md`
  - `delivery_constraints.md`

Agent 在进入模板选择与故事线规划前，必须先能用一句话回答：

- 这份 PPT 是为谁做的
- 要达成什么目标
- 最重要的 3 个展示重点是什么
