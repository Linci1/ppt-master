# 长亭安服 QA 质量护栏

## 一、护栏目标

这个 QA 配置用于 `security_service` 模板，对应长亭安服历史胶片风格。

检查过程必须同时拦住两类问题：

- 布局缺陷
- 历史风格偏移

如果当前页面属于复杂正文页（如体系图、攻击链、闭环图、泳道图、矩阵图、证据墙），还必须同时对照 `advanced_page_patterns.md` 检查图形逻辑、呈现效果和文图协同是否成立。

对攻击链、复杂案例链、证据证明页、治理矩阵页，还必须额外对照：

- `complex_graph_semantics.md`
- `complex_case_chain_modeling.md`
- `complex_page_logic_qa_checklist.md`
- `evidence_grading_rules.md`
- `complex_deck_orchestration.md`
- `complex_svg_blueprints.md`
- `sample_grade_content_system.md`
- `soft_content_qa_framework.md`
- `soft_content_rewrite_strategies.md`

若复杂页只满足“内容多、结构多”，但不满足这些文档中的语义、关系、证据与主判断要求，仍应判定为未通过。

---

## 二、模板身份检查

在逐页检查前，先确认整份 PPT 仍然像一份长亭安服演示，而不是普通项目汇报。

### 必须保留

- 蓝橙浅底的品牌方向
- 固定骨架：顶部蓝条、左侧蓝竖条、标题引导条、底部橙条、页码区
- 长亭品牌 Logo，且必须使用批准版本
- 章节节奏：理念 / 体系 / `攻` / `防` / `培` / 运营 / 案例 / 背书
- 结论先行、证据导向的表达习惯

### 必须避免

- 退化成普通项目汇报
- 退化成咨询报告式长段文字页
- 退化成只会堆卡片的模板化输出
- 漏放 Logo、错放 Logo、浅底页继续使用白色 Logo、或自行给 Logo 加底板

---

## 三、固定骨架规则

### 保护区

| 区域 | 规则 |
|------|------|
| 标题安全区 | 标题与引导条不可被正文压住 |
| 顶部 Logo 安全区 | 标题、图示、标签不得侵入 |
| 页脚保护区 | 正文与图表不得侵入页码区 |
| 底部装饰区 | 内容不得贴底部橙条 |
| 封面 Logo 区 | 不得与封面 Logo 重叠 |
| 结束页 Logo 区 | 不得与结束页 Logo 重叠 |

### 骨架不变量

- 顶部蓝条必须保持完整
- 左侧蓝竖条属于模板骨架，不计入正文区
- 标题栏下方应保留足够留白
- 页码只承担页码职责，不承载正文补充信息

---

## 四、布局 QA

每一页都必须检查以下问题。

### 文本质量

- 中文断句是否自然
- 文本是否出现明显 AI 味长句
- 标题、副标题、标签是否存在孤字换行

### 贴边与拥挤

- 文本是否贴边
- 卡片内四周留白是否不足
- 标题与分隔线是否过近

### 溢出与重叠

- 卡片内文本是否视觉溢出
- 不同模块之间是否重叠
- 图示标签是否压线或互相打架
- Logo / 页码 / 底饰条是否与正文冲突

### 信息密度

- 每页信息密度是否过高
- 是否出现为了塞内容而压缩字号或行距
- 是否应该拆页，而不是硬塞

---

## 五、安服文本与论证 QA

`security_service` 不只是检查“排得下”，还必须检查“讲得对”。

对安服 / HW / 攻防演练 / 安全运营类正文页，文本必须从“原文缩写”升级为“证据驱动的判断表达”。

### 生成前强制字段

每个正文页在进入 SVG 生成前，必须先明确以下 4 项：

- `页面意图`：这页给谁看，要解决什么判断问题
- `证明目标`：这页最终要证明什么
- `核心判断`：一句话主结论
- `支撑证据`：2-5 条支撑这个判断的事实

若其中任意一项模糊不清，这页不得直接进入排版。

### 软性内容风险检查

除硬性布局与文本规则外，还必须拦截以下软性问题：

- 标题、模块标题、页尾判断彼此不在同一条逻辑线上
- 模块只是并列摆放，但没有说明彼此关系
- 证据存在，却没有承担明确证明作用
- 图和文同时存在，但在表达两个不同层面的内容
- 页面写了很多动作，却没有形成治理判断或动作闭环
- 页面只是复杂整理，不是复杂论证

若命中以上任一情况，即使页面没有排版错误，也应视为未通过。

### 软性问题修复规则

若命中软性内容风险，不应只改局部词句，而应按以下顺序重写：

- 先收敛唯一主判断
- 再明确模块关系类型
- 再统一标题、模块标题、页尾判断的逻辑层级
- 再重绑证据到节点 / 关系 / 结果
- 最后重排模块顺序与视觉主次
- 重写动作统一以 `soft_content_rewrite_strategies.md` 为执行基线

如果页面仍需依赖大量讲解才能成立，则视为重写未完成。

### 单页文本判定标准

每一页都必须满足：

- 只有 1 个主判断，不允许一页并列多个中心结论
- 标题能准确代表主判断，而不是空泛栏目名
- 各模块都在支撑主判断，而不是各说各话
- 原始截图、数据或案例必须承担“证明作用”，不能只做配图
- 结论条、建议条必须是判断句或动作句，不能只是描述句

### 推荐文本逻辑类型

正文页应优先命中以下逻辑类型之一：

- 风险总览型
- 攻击链证明型
- 案例拆解型
- 问题机制型
- 治理整改型
- 管理摘要型

若页面内容复杂，但以上类型无法判断，说明内容还没有被提炼到可生成状态。

### 语言表达规则

推荐表达：

- 先结论，再展开
- 先讲风险如何形成，再讲现象
- 先讲链路和影响，再讲零散问题
- 多用判断句、动作句、机制句、证据句

不推荐表达：

- 长段自然段说明
- 空泛判断，如“风险较大”“需要重视”
- 报告原句机械压缩
- 介绍式、说明式、流水账式写法

### 术语白名单

优先使用以下表达：

- 攻击入口
- 突破链
- 管理面
- 高价值资产
- 横向扩展
- 关键证据
- 证明点
- 控制矩阵
- 整改闭环
- 复测验证
- 残余风险
- 优先级
- 放大机制

### 术语黑名单

命中以下表达时，应直接判定为需要重写：

- 证据驾驶舱
- 证据挂载
- 挂载型案例页
- 高级洞察
- 一体化协同提效
- 能力沉淀闭环体系
- 安全水位拉齐
- 攻击赋能

### 句式检查

每页至少应包含以下句式中的一种：

- `判断句`：例如“该结果已足以支撑稳定突破链成立”
- `动作句`：例如“应优先收口外网入口并同步整改管理面”
- `机制句`：例如“外网问题与管理面暴露叠加后，会显著降低横向成本”
- `证据句`：例如“原始截图已证明后台、数据库与云侧凭证均可直接获取”

若页面只有描述句、没有判断句，视为不合格。

### 高信息密度页文本压缩规则

复杂页必须同时满足：

- 一页只允许 1 个主判断
- 主判断最多拆成 3 个分判断
- 每个分判断最多挂 2-3 条证据
- 模块标题尽量控制在 6-12 字
- 结论条尽量控制在 18-34 字
- 模块内正文尽量控制在 24-42 字一条

禁止通过无限缩字号、压行距、堆解释句的方式容纳内容。

### 案例级复杂正文检查

命中复杂页后，还必须追加以下检查：

- 标题、节点、证据说明、页尾判断是否共享同一条 `论证主线`
- 每个主证据是否都存在明确锚点：节点 / 关系 / 结果 / 分判断
- 页面是否只承担一个主复杂任务，而不是同时解释两套结构
- 文案是否已经压缩成顾问判断，而不是报告原句的删减版
- 相邻复杂页是否形成推进关系，而不是重复证明

只要以上任一项不成立，即使页面排版工整，也不能视为通过。

### 图文协同检查

复杂页必须明确图文分工：

- 图负责结构、关系、链路、分层
- 文负责判断、解释、结论、建议

因此要重点检查：

- 图旁文本是否只是在重复节点文字
- 原始证据图旁是否存在明确“证明句”
- 结构图外是否存在一句管理判断或价值判断
- 是否出现“图很复杂，但读者不知道这图想证明什么”

## 六、历史风格 QA

### 叙事检查

- 当前页是否处于正确章节语境
- 标题是否有结论感、价值表达
- 页面是否在“证明”，而不是“描述”
- 整份 PPT 是否仍然符合长亭安服胶片节奏

### 图示检查

- 体系页是否表达了“平台 + 服务 + 结果”的关系
- `攻 / 防 / 培 / 运营` 页是否使用了能力地图或流程结构
- 运营页是否具备闭环感、链路感
- 案例页是否包含背景、动作、结果中的至少三类信息
- 背书页是否形成证据墙，而不是普通罗列
- 页型选择是否与 `advanced_page_patterns.md` 中定义的高级正文模式匹配，而不是错用普通页型承载复杂内容

### 风格漂移信号

出现以下任意一种情况，都应判定为风格偏移：

- 重要能力页被降级成普通列表页
- 案例页没有结果 headline
- 背书页没有证据结构
- 多页都只有文本块，缺少图示支撑

## 七、分页面密度阈值

| 页面类型 | 阈值 |
|----------|------|
| `17_service_overview.svg` | 最多 3 个域能力 + 1 个平台区 + 1 个价值区，不再继续塞次级模块 |
| `18_domain_capability_map.svg` | 最多 4 个能力模块；每模块 1 行标题 + 1-2 行说明 |
| `19_result_leading_case.svg` | 每个栏目最多 3 条要点；必须保留 headline 与 KPI 空间 |
| `07_data.svg` | KPI 数值尽量短；说明项最多 3 条 |
| `13_highlight.svg` | 侧边统计标签必须短，不能承载长中文短语 |
| `16_table.svg` | 表格说明列与右侧发现区不得同时写满，必要时拆页 |
| `12_grid.svg` | 一页最多 6 个卡片，但每卡只放 1 个核心点 |

---

## 八、修复优先级

发现问题后，按以下顺序修复：

1. Logo / 页脚 / 页码冲突
2. 文本溢出 / 重叠
3. 主判断缺失 / 图文不协同 / 术语失真
4. 信息密度超载
5. 历史风格偏移
6. 局部对齐与间距打磨

---

## 九、导出前判定标准

只有同时满足以下条件，使用 `security_service` 的成品才允许导出：

- 没有严重布局冲突
- 没有明显文本溢出
- 每一页都保留了批准的长亭 Logo 资产，且版本与背景匹配
- 每一页正文都能说清页面意图、证明目标与核心判断
- 复杂页不存在术语黑名单表达，不存在明显 AI 味标题或标签
- 原图页、案例页、攻击链页都能形成“证据 → 判断”关系
- 关键页面仍符合历史胶片的叙事与图示习惯
- 案例页保持结果导向
- 背书页仍然呈现为证据页
