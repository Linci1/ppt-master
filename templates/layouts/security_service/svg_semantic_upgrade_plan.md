# 长亭安服关键页型占位语义升级方案

> 目标：在正式修改 SVG 视觉骨架之前，先明确 `security_service` 中 4 个关键页型应当承接什么复杂结构、各占位区分别承担什么信息，以及哪些复杂表达不应继续用“普通说明页语义”去硬塞。

> 适用对象：
> - `05_case.svg`
> - `17_service_overview.svg`
> - `18_domain_capability_map.svg`
> - `19_result_leading_case.svg`

---

## 一、升级原则

当前这 4 个页型已经具备基础骨架，但占位语义还偏向：

- 单块说明
- 普通案例摘要
- 常规能力拆分

而后续希望承接的是：

- 复杂体系图
- 攻击链 / 证明链
- 闭环机制图
- 泳道协同图
- 高密度专业案例页

所以升级重点不是先改颜色或边框，而是先改“信息槽位的角色定义”。

### 1.1 语义升级的核心目标

- 让 Strategist 更容易把复杂内容映射到正确页型
- 让 Executor 明确每个文本槽位不是“随便填文案”，而是承接特定结构角色
- 让后续 SVG 改造可以围绕语义进行，而不是围绕视觉拍脑袋调整

### 1.2 统一约束

这 4 类页即使升级为复杂正文页，也仍需遵守：

- 固定页眉、页脚、Logo 安全区不变
- 正文主结构优先于装饰变化
- 复杂感来自结构和关系，不来自无序堆砌

## 二、`17_service_overview.svg` 升级方案

### 2.1 当前问题

当前槽位大致是：

- `OVERVIEW_LEAD`
- `PLATFORM_NAME`
- `PLATFORM_DESC`
- `DOMAIN_ATTACK_*`
- `DOMAIN_DEFENSE_*`
- `DOMAIN_TRAINING_*`
- `VALUE_1-3`
- `DRIVER_TITLE`
- `DRIVER_POINT_1-2`

它适合基础“体系概览页”，但不够适合更复杂的：

- 多层服务体系
- 平台 + 域能力 + 输出结果
- 中央引擎 + 左右能力域 + 下层驱动机制

### 2.2 升级后的页面定位

定位为：

- **分层体系图主承接页**
- 也可承接轻量闭环机制页

### 2.3 建议语义角色

| 当前槽位 | 建议升级语义 | 用途 |
|---------|-------------|------|
| `OVERVIEW_LEAD` | `体系总引句` | 一句话定义整页想证明什么 |
| `PLATFORM_NAME` | `中央引擎/平台名称` | 页面中心骨架 |
| `PLATFORM_DESC` | `中央引擎说明` | 平台职责，不超过 2 行 |
| `DOMAIN_ATTACK_*` | `一级域 A` | 可泛化为攻/防/运营/治理中的任一主域 |
| `DOMAIN_DEFENSE_*` | `一级域 B` | 同上 |
| `DOMAIN_TRAINING_*` | `一级域 C` | 同上 |
| `VALUE_1-3` | `结果/价值输出` | 证明体系最终产出什么 |
| `DRIVER_TITLE` | `底层驱动机制标题` | 方法论、数据、平台、运营驱动 |
| `DRIVER_POINT_1-2` | `底层驱动机制要点` | 驱动因素，不是普通注释 |

### 2.4 推荐承接结构

优先承接：

1. `中央引擎 -> 一级域 -> 结果输出`
2. `平台 -> 服务模块 -> 客户价值`
3. `方法论核心 -> 三大能力域 -> 最终成效`

不推荐承接：

- 需要 4 个以上一级域的复杂地图
- 需要多角色泳道的协同流程
- 需要明确攻击链节点的案例页

### 2.5 后续 SVG 改造建议

- 强化“中央引擎”视觉中心
- 左侧平台区、中央能力区、右侧结果区做得更明确
- 下方驱动条更像“底座”，不要只是两个普通 bullet

## 三、`18_domain_capability_map.svg` 升级方案

### 3.1 当前问题

当前槽位大致是：

- `DOMAIN_LABEL`
- `SCENE_TITLE`
- `SCENE_POINT_1-3`
- `CAPABILITY_1-4`
- `OUTCOME_TITLE`
- `OUTCOME_1-3`
- `METHOD_NOTE`

它已经接近复杂正文页，但当前更像：

- 左侧场景
- 中间四能力
- 右侧结果

还不够明确地支持：

- 闭环机制
- 成熟度模型
- 场景到能力再到结果的复杂推导页

### 3.2 升级后的页面定位

定位为：

- **闭环运营 / 能力地图主承接页**
- 兼容成熟度模型、方法论映射图

### 3.3 建议语义角色

| 当前槽位 | 建议升级语义 | 用途 |
|---------|-------------|------|
| `DOMAIN_LABEL` | `主域标签` | 攻 / 防 / 培 / 运营，或任一一级主题 |
| `SCENE_TITLE` | `场景/问题定义` | 解释该域面对什么现实场景 |
| `SCENE_POINT_1-3` | `场景约束/关键挑战` | 左侧输入，不是普通描述 |
| `CAPABILITY_1-4` | `核心能力模块` | 中央主结构，承接能力、环节、控制点 |
| `OUTCOME_TITLE` | `输出结果标题` | 右侧结果区总标题 |
| `OUTCOME_1-3` | `结果/收益/证明点` | 结果不是动作，而是输出 |
| `METHOD_NOTE` | `方法论补充/机制说明` | 页底方法注，不是尾注 |

### 3.4 推荐承接结构

优先承接：

1. `场景 -> 核心能力 -> 结果`
2. `问题域 -> 控制点 -> 成效`
3. `输入 -> 四环节闭环 -> 输出`
4. `成熟度维度 -> 能力模块 -> 建设目标`

不推荐承接：

- 时间线型页面
- 需要多张证据图附着的案例链页面

### 3.5 后续 SVG 改造建议

- 中间四块能力区需要更像“结构骨架”，而不是 4 个并列小卡片
- 左侧场景区和右侧结果区建议进一步强化输入 / 输出对照感
- 页底 `METHOD_NOTE` 建议升级为“方法论条”，而不是一行普通说明

## 四、`19_result_leading_case.svg` 升级方案

### 4.1 当前问题

当前槽位大致是：

- `CLIENT_CONTEXT`
- `RESULT_HEADLINE`
- `KEY_METRIC`
- `KEY_METRIC_LABEL`
- `CHALLENGE_1-3`
- `ACTION_1-3`
- `RESULT_1-3`

这已经是不错的结果导向案例页，但它目前偏：

- 三列表达

还不足以完全承接：

- 攻击链案例
- 防守链案例
- “路径型”案例
- 更复杂的证明链页面

### 4.2 升级后的页面定位

定位为：

- **结果导向案例主承接页**
- 可兼容轻量攻击链、轻量证明链

### 4.3 建议语义角色

| 当前槽位 | 建议升级语义 | 用途 |
|---------|-------------|------|
| `CLIENT_CONTEXT` | `客户/场景上下文` | 行业、客户属性、问题背景 |
| `RESULT_HEADLINE` | `结果 headline` | 页面真正要证明的主结论 |
| `KEY_METRIC` | `单一核心指标` | 结果锚点 |
| `KEY_METRIC_LABEL` | `指标说明` | 指标是什么，不宜过长 |
| `CHALLENGE_1-3` | `关键挑战 / 关键前提` | 左列输入条件 |
| `ACTION_1-3` | `关键动作 / 关键路径` | 中列动作链 |
| `RESULT_1-3` | `结果证明 / 价值产出` | 右列产出链 |

### 4.4 推荐承接结构

优先承接：

1. `挑战 -> 动作 -> 结果`
2. `前提 -> 方案 -> 成效`
3. `攻击条件 -> 突破动作 -> 最终影响`
4. `客户痛点 -> 服务动作 -> 改善结果`

不推荐承接：

- 节点很多的重型攻击链
- 需要展示大量截图证据的案例页

### 4.5 后续 SVG 改造建议

- 保持 headline 和 KPI 作为顶部最强焦点
- 三列应强化“因果链”，而不是并排摆放
- 允许在列内增加节点图标或小连线，增强链路感

## 五、`05_case.svg` 升级方案

### 5.1 当前问题

当前槽位大致是：

- `CASE_BACKGROUND`
- `CASE_SOLUTION`
- `CASE_PROCESS`
- `CASE_RESULTS`
- `CASE_IMAGE`
- `CASE_CLIENT`

这是典型“传统案例页”语义：

- 背景
- 方案
- 过程
- 结果

它适合普通案例页，但不足以支撑：

- 泳道协同页
- 复杂实施页
- 多阶段协同案例页
- 轻量路径图

### 5.2 升级后的页面定位

定位为：

- **通用复杂案例页 / 泳道协同主承接页**

### 5.3 建议语义角色

| 当前槽位 | 建议升级语义 | 用途 |
|---------|-------------|------|
| `CASE_BACKGROUND` | `背景 / 输入条件` | 问题、上下文、现状 |
| `CASE_SOLUTION` | `角色或方案主线 A` | 可解释为客户侧 / 平台侧 / 主方案 |
| `CASE_PROCESS` | `角色或方案主线 B` | 可解释为长亭侧 / 执行侧 / 阶段链 |
| `CASE_RESULTS` | `结果 / 交付 / 收束` | 最终产出与价值 |
| `CASE_IMAGE` | `结构图 / 证据图 / 泳道辅助图` | 不只是插图占位 |
| `CASE_CLIENT` | `客户标签 / 页尾补充语义` | 客户属性、项目标签、案例编号 |

### 5.4 推荐承接结构

优先承接：

1. `背景 -> 协同动作 -> 结果`
2. `客户侧 -> 长亭侧 -> 联动成果`
3. `战前 / 战中 / 战后` 的轻量流程页
4. `实施步骤 + 交付结果` 的服务页

不推荐承接：

- 需要中心平台骨架的体系图
- 需要强结果 headline 的结果页

### 5.5 后续 SVG 改造建议

- 左上和右上两块不应只是“背景/方案”静态四宫格
- 需要允许其升级为：
  - 双泳道
  - 上下阶段
  - 协同结构
- 中下部图片区建议从“图片占位”升级成“结构辅助图 / 证据区”

## 六、四个关键页型的分工边界

为了避免后续页型继续混用，建议明确边界：

| 页型 | 最适合承接 | 不宜承接 |
|------|-----------|---------|
| `17_service_overview.svg` | 体系总览、平台支撑、多层结构 | 重型案例链、泳道协同 |
| `18_domain_capability_map.svg` | 闭环、能力地图、成熟度模型 | 时间线、重证据案例 |
| `19_result_leading_case.svg` | 结果导向案例、轻量证明链 | 大量截图附着的攻击链 |
| `05_case.svg` | 协同型案例、通用复杂案例、轻量流程 | 体系图、强 headline 结果页 |

---

## 七、后续实施顺序建议

建议按以下顺序推进后续 SVG 真正改造：

1. 先改 `17_service_overview.svg`
   - 体系图是最基础、复用率最高的复杂页
2. 再改 `18_domain_capability_map.svg`
   - 它决定了闭环图、能力图、成熟度图的专业度
3. 再改 `19_result_leading_case.svg`
   - 拉升案例页的结果导向和证明感
4. 最后改 `05_case.svg`
   - 让通用案例页承接协同和轻量流程能力

---

## 八、结论

这 4 个 SVG 不应再被理解成：

- 一个体系页
- 一个能力页
- 一个案例页
- 一个普通案例页

而应分别理解成：

- `17_service_overview.svg`：**体系骨架页**
- `18_domain_capability_map.svg`：**闭环/能力结构页**
- `19_result_leading_case.svg`：**结果证明页**
- `05_case.svg`：**协同/实施案例页**

只有先把这个语义层定义清楚，后续改 SVG 视觉和生成逻辑时才不会反复跑偏。
