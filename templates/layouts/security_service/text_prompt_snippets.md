# 长亭安服文本生成 Prompt 片段

> 用途：给 `security_service` 的 Strategist / Executor 作为可直接拼接的提示片段，约束安服正文页的文本提炼方式。

---

## 一、通用总提示

```text
你正在为长亭安服模板生成正文内容。

这不是普通汇报页，也不是漏洞清单缩写页。你必须把源材料中的事实，提炼成“证据驱动的判断表达”。

请始终遵循以下规则：
1. 每页先明确 页面意图 / 证明目标 / 核心判断 / 支撑证据
2. 每页只能有 1 个主判断
3. 先结论，再展开；先证明，再描述
4. 图负责结构，文负责判断、解释和结论
5. 不要写成长段说明，不要堆空话，不要使用 AI 自造术语

禁止使用以下表达：
- 证据驾驶舱
- 证据挂载
- 挂载型案例页
- 高级洞察
- 一体化协同提效
- 能力沉淀闭环体系

优先使用以下表达：
- 风险统计总览
- 攻击路径分析
- 内网突破案例链
- 关键证据 / 管理摘要
- 人员侧攻击结果
- 控制矩阵 / 清理闭环
- 整改优先级与闭环
```

---

## 二、软性问题重写提示

```text
如果页面已经有基本结构，但仍存在“有其表、无其实”的软性问题，不要只改几个词。

请按以下顺序重写：
1. 先收敛唯一主判断
2. 再明确模块关系：并列 / 因果 / 放大 / 汇聚 / 闭环
3. 再重写标题、模块标题、页尾判断，让三者在同一条线
4. 再把每个证据绑到节点 / 关系 / 结果
5. 最后删掉只是显得复杂、却不能提升理解效率的内容

重写时重点改 5 个部位：
- 标题：改成主判断
- 模块标题：改成分判断或机制判断
- 证据说明：改成“该图证明什么”
- 页尾收束：改成管理判断 / 动作要求 / 下一页承接
- 模块顺序：按主线重排，而不是沿用原文顺序
```

## 三、软性逻辑自检提示

```text
在开始写页面正文前，先用以下 6 个问题做一次软性自检：
1. 这页唯一主判断是什么？
2. 模块之间是并列、因果、放大、汇聚还是闭环？
3. 标题、模块标题、页尾判断是不是同一条线？
4. 每个证据分别证明哪个节点 / 关系 / 结果？
5. 图和文是在互相增强，还是在互相重复？
6. 这页最后要把读者带到什么管理判断或动作要求？

如果有任一问题答不上来，不要直接写正文，先重组页面逻辑。
```

## 四、单页提炼提示

```text
请先不要直接写页面正文。

先输出以下 4 项：
1. 页面意图
2. 证明目标
3. 核心判断（1 句）
4. 支撑证据（2-5 条）

然后基于这 4 项，再生成页面正文文本。

要求：
- 标题必须能代表核心判断
- 模块标题必须是小结论，不是空栏目名
- 结论条必须是判断句或动作句
- 若存在原始截图，必须补一句“该图证明什么”
```

## 五、案例级复杂页论证主线提示

```text
当前页面需要达到更接近历史案例胶片的复杂正文水位。

请先不要画图，也不要直接写正文。
先把原文压缩成一条 `论证主线`：
1. 现象 / 入口
2. 放大机制 / 关键条件
3. 结果 / 影响
4. 管理判断 / 动作要求

然后检查以下四件事：
- 标题是否能代表整条论证主线
- 主图是否在展示这条主线，而不是平铺信息
- 证据是否都能锚定到节点、关系或结果
- 页尾判断是否在收束这条主线

如果这四件事有任何一件答不上来，说明这页还不能进入 SVG 生成。
```

## 六、案例级文本压缩提示

```text
请把原文从“报告叙述”压缩成“安服判断表达”。

压缩顺序固定为：
- 先保留结论和机制
- 再保留关键动作和结果
- 最后才保留背景描述

请分别产出：
1. 标题（1 句判断）
2. 节点文案（对象 + 动作 / 状态变化）
3. 证据说明（该图证明什么）
4. 页尾收束（这页最终说明了什么）

禁止：
- 中性栏目名
- 项目过程流水账
- 为了显得高级而自造术语
- 大段说明替代判断
```

## 七、复杂逻辑页提示

```text
当前页面属于复杂正文页。你必须先判断它属于以下哪一类：
- 风险总览型
- 攻击链证明型
- 案例拆解型
- 问题机制型
- 治理整改型
- 管理摘要型

确定逻辑类型后，再组织内容。

复杂页必须满足：
- 1 个主判断
- 最多 3 个分判断
- 每个分判断最多 2-3 条证据
- 标题、图形、结论条都服务于同一个主判断
- 不允许只罗列事实，不给判断
```

---

## 八、攻击链 / 复杂链路页提示

```text
请把原始内容提炼成“攻击链证明页”。

文本要覆盖：
1. 攻击入口
2. 关键动作
3. 放大条件
4. 结果层
5. 管理判断

推荐语气：
- 该链路已具备可复现条件
- 风险不在单点漏洞，而在入口、管理面与凭证条件的串联
- 当前问题已足以支撑稳定突破链成立

不要写成漏洞列表，也不要只描述过程。
```

---

## 九、案例页提示

```text
请把原始内容提炼成“案例拆解页”。

正文建议固定为以下四段：
- 触达：攻击方如何进入场景
- 动作：做了什么
- 结果：拿到了什么结果
- 证明点：这说明了什么

要求：
- 标题要有结果导向
- 每个案例卡片必须有一句证明点
- 页面底部必须有总判断
- 不要写成项目过程流水账
```

---

## 十、治理整改页提示

```text
请把原始内容提炼成“治理整改页”。

正文必须同时包含：
- 优先级（P1 / P2 / P3）
- 关键动作
- 验证方式
- 完成标准

推荐表达：
- P1 优先收口可直接形成落点的问题
- 完成标准不是删除痕迹，而是完成复测、复核与闭环留痕
- 文件、内存、终端、账号、日志必须一起闭环
```

---

## 十一、高信息密度页压缩提示

```text
当前页面允许高信息密度，但不允许高混乱度。

请压缩内容时遵循：
- 删掉重复解释
- 删掉无效背景
- 把长句改成判断句
- 把并列说明合并成机制句
- 优先保留：入口、动作、放大条件、结果、建议

字数控制建议：
- 模块标题：6-12 字
- 结论条：18-34 字
- 模块正文：单条 24-42 字
```
