# 长亭安服高级正文页模式

> 目标：把 `security_service` 模板从“稳定的安服胶片骨架”升级为“能够稳定承载复杂图形、复杂流程图和高密度专业内容”的正文体系。
>
> 前提约束：这里的“高级正文页”不是为了视觉上更复杂，而是为了更准确地服务源文档。如果文档内容本身并不复杂，或简单结构已经足以说明问题，则不应强行命中高级正文模式。

> 这份文档不只定义“页面长什么样”，更定义：
> - 页面为什么这样组织
> - 图形为什么这样画
> - 文本如何与图形配合
> - 哪些复杂页适合进入长亭安服模板

> 机器可读映射已同步写入 `templates/layouts/layouts_index.json` 的 `security_service.advancedPageStrategy`，便于 Strategist 先判模式、再选页型。

---

## 一、分析来源

本页型抽象主要参考以下两份历史材料：

- `长亭安服主打胶片- v2.2-0427.pptx`
- `2025HW总结-25.09.004.pptx`

两者分别提供不同价值：

- `长亭安服主打胶片`：提供安服胶片的稳定章节节奏、结论先行标题、能力证明型表达
- `2025HW总结`：提供复杂知识页、重型流程页、攻击链、运营闭环、矩阵关系图等高密度正文能力

这意味着后续生成 `security_service` 时，不能只复用历史模板骨架，也要复用其中成熟的“复杂页表达语法”。

## 二、抽象维度

后续所有高级正文页，都应同时从四个维度定义。

### 2.1 文本逻辑

回答：

- 这页要证明什么
- 标题、一级模块、二级模块、指标、结论之间是什么关系
- 内容是并列、递进、因果、闭环、映射还是链路

### 2.2 图形绘画逻辑

回答：

- 为什么应该画成分层图、闭环图、泳道图、攻击链、矩阵或证据墙
- 主结构是什么
- 节点、层级、连线和分区如何组织

### 2.3 呈现效果逻辑

回答：

- 页面视觉焦点在哪里
- 阅读顺序如何流动
- 高密度内容如何保持稳、清晰、专业

### 2.4 文图协同逻辑

回答：

- 图负责结构，文负责解释
- 还是图负责证据，文负责结论
- 或者图负责流程，文负责阶段定义

> 对于 `security_service` 高级正文页，推荐采用 **中耦合文图协同** 为主、**局部高耦合** 为辅：
> - 中耦合：图形承担结构，文本承担解释
> - 高耦合：图形节点与文字说明强绑定，常见于攻击链、流程图、矩阵图

### 选型原则补充

- **主承接页型**：优先选用，除非内容结构与模板骨架明显不匹配
- **次承接页型**：内容意图一致，但主结构略有差异时使用
- **兜底页型**：只有在现有专用页型都无法承载时才使用

### 明确禁用

以下情况即使可以“塞进去”，也不应选错页型：

- 体系图页退化成 `11_list.svg`
- 攻击链页退化成 `07_data.svg`
- 闭环机制页退化成普通对比页
- 证据墙页退化成纯 logo 拼贴或 `03_content.svg` 普通排版
- 多泳道实施链退化成两列静态对照
- 攻击树页退化成普通 6 条 bullet
- 数据证明页退化成 KPI 卡片 + 一张普通图表
- 治理看板退化成没有结构重点的普通数据表
- 复杂案例链退化成“挑战 / 动作 / 结果”三栏静态并列

## 1. 分层体系图

### 页面意图

用于表达：

- 安服服务体系
- 平台支撑体系
- 多层能力架构
- 数据、平台、能力、运营之间的依赖关系

### 文本逻辑

- 页面标题必须是结论式体系标题
- 页面中心应存在一个“总平台 / 总引擎 / 总方法论”
- 往下拆分 3-5 个一级域
- 每个一级域再附带一句职责或价值说明

推荐语法：

```text
总引擎 / 总平台
→ 一级域 A / B / C / D
→ 每域对应的关键动作 / 核心价值
```

### 图形绘画逻辑

- 优先使用纵向分层或中心辐射结构
- 不适合做成普通多卡片平铺
- 每层之间的关系应该靠位置体现，而不是靠大段说明文字
- 一级域数量建议控制在 3-5 个

### 呈现效果

- 视觉重点是“稳”“全”“有总有分”
- 重心应位于页面中部
- 蓝色结构层负责体系骨架，橙色只强调结果或关键收益

### 文图协同

- 图形负责层级和关系
- 文本负责命名、定义、价值说明
- 每个域内的说明文字以 1-2 行为宜

### 推荐页型

- `17_service_overview.svg`
- 次选：`08_product.svg`
- 兜底：深度改造后的 `03_content.svg`

### 高风险点

- 为了塞更多模块，把层内说明写成长段落
- 把体系图退化成普通四卡片
- 一级域过多导致重心失稳

## 2. 阶段演进 / 路线图

### 页面意图

用于表达：

- 规则演进
- 能力建设路线
- 安服成熟度演进
- 年度趋势变化

### 文本逻辑

- 时间或阶段是主轴
- 每个阶段都要包含：
  - 阶段名
  - 核心变化
  - 影响 / 应对

推荐语法：

```text
阶段 1：发生了什么
阶段 2：变化了什么
阶段 3：为什么重要
阶段 4：我方如何应对
```

### 图形绘画逻辑

- 优先用横向阶段带，而不是零散时间点
- 每一阶段是一个完整块，不只是一个圆点
- 重要阶段需要用高亮色强调

### 呈现效果

- 阅读方向应非常明确，从左到右推进
- 适合做成“主时间轴 + 下方解释块”
- 视觉上应有节奏感，不应像均匀平铺表格

### 文图协同

- 图负责演进顺序
- 文负责阶段含义和影响解释
- 如需额外分析，放侧边总结框，不要塞进主轴内部

### 推荐页型

- `10_timeline.svg`
- 次选：`15_section.svg`, `09_comparison.svg`
- 兜底：深度改造后的 `03_content.svg`

### 高风险点

- 时间轴只剩年份，没有变化逻辑
- 每个阶段文案长度差距过大
- 所有阶段视觉同权，缺少重点

## 3. 攻击链 / 案例链路图

### 页面意图

用于表达：

- 入侵路径
- 攻击案例
- 横向移动链路
- 供应链迂回
- 大模型攻击链

### 文本逻辑

必须形成链路，而不是普通案例摘要。

推荐语法：

```text
起点 / 前提
→ 关键动作
→ 中间突破
→ 靶标达成
→ 最终结果 / 影响
```

可附加：

- 攻击条件
- 关键证据
- 薄弱点
- 防守失效点

### 图形绘画逻辑

- 优先横向链、折线链或分叉链
- 每个节点必须是动作或状态节点
- 截图、证据、旁注要附着在节点附近，而不是满页散落

### 呈现效果

- 必须有推进感和压迫感
- 突破节点、关键转折点要明显突出
- 节点顺序必须一眼可追踪

### 文图协同

- 图形负责路径
- 文本负责节点含义、前提条件、结果说明
- 单页建议控制在 6-8 个关键节点

### 推荐页型

- `19_result_leading_case.svg`
- 次选：`05_case.svg`
- 兜底：深度改造后的 `03_content.svg`

### 高风险点

- 用一堆截图拼出“复杂感”，但没有路径主线
- 案例只写背景和结果，中间动作缺失
- 节点过多、连线交叉、文字压线

---

## 4. 闭环运营 / 机制图

### 页面意图

用于表达：

- 安全运营闭环
- 风险评估闭环
- 平战切换机制
- 持续优化体系

### 文本逻辑

它不是一次性流程，而是循环机制。

推荐语法：

```text
输入
→ 分析
→ 动作
→ 反馈
→ 优化
→ 再进入下一轮
```

### 图形绘画逻辑

- 优先环形、半环形、螺旋形
- 页面中心宜放平台 / 引擎 / 机制核心
- 外圈放动作或阶段节点

### 呈现效果

- 必须有“循环感”和“机制感”
- 不能画成普通线性流程
- 适合用于安全运营、持续优化、体系化运营

### 文图协同

- 图形承担循环结构
- 文本承担节点定义和少量价值说明
- 长解释优先放到侧边说明区

### 推荐页型

- `18_domain_capability_map.svg`
- 次选：`17_service_overview.svg`, `06_tactics.svg`
- 兜底：深度改造后的 `03_content.svg`

### 高风险点

- 画成四个并列卡片却说自己是闭环
- 节点定义过长，导致环内拥挤
- 结果区不突出，看不出闭环最终价值

## 5. 泳道协同图

### 页面意图

用于表达：

- 长亭侧 / 客户侧协同
- 平台侧 / 租户侧协同
- 战前 / 战中 / 战后联动
- 多角色处置协同

### 文本逻辑

同时包含两个维度：

- 一个维度是阶段
- 一个维度是角色 / 责任主体

推荐语法：

```text
阶段轴：准备 → 执行 → 响应 → 收束
角色轴：客户侧 / 长亭侧 / 平台侧 / 运营侧
每个交叉点填入动作、输出或责任
```

### 图形绘画逻辑

- 横向为阶段，纵向为角色
- 阶段列必须统一对齐
- 泳道内写动作和交付，不写大段解释

### 呈现效果

- 应呈现高专业度和协作感
- 页面看上去“复杂但不乱”
- 非常适合表达实施类和运营类服务

### 文图协同

- 图形负责责任划分
- 文本负责动作、交付物、协作结果
- 最右侧可增加结果汇总区

### 推荐页型

- `05_case.svg`
- 次选：`10_timeline.svg`, `06_tactics.svg`
- 兜底：深度改造后的 `03_content.svg`
- 如果后续新增专用 SVG，优先补泳道模板

### 高风险点

- 把泳道图做成表格
- 每个格子里塞过多解释性文字
- 阶段和角色都很多，导致页面密度失控

## 6. 矩阵防御 / 映射图

### 页面意图

用于表达：

- 风险域与控制域映射
- API 安全矩阵
- 漏洞治理矩阵
- 资产 / 风险 / 响应 / 价值关系

### 文本逻辑

必须至少有两个交叉维度。

推荐语法：

```text
维度 A：风险类型 / 资产对象 / 能力域
维度 B：控制点 / 响应动作 / 服务模块
交叉区：关键措施 / 指标 / 结果
```

### 图形绘画逻辑

- 优先做“矩阵 + 局部高亮”
- 不要退化成普通表格
- 重点交叉区需要视觉强调

### 呈现效果

- 应有明显的面状结构
- 页面要显得专业、精密，而不是死板
- 适合蓝色框架 + 橙色重点交叉点

### 文图协同

- 图形承担多维关系
- 文本承担单元定义和重点解释
- 单元格内尽量只写短标签，不写多句描述

### 推荐页型

- `12_grid.svg`
- 次选：`16_table.svg`, `09_comparison.svg`
- 兜底：深度改造后的 `03_content.svg`

### 高风险点

- 单元格内文字过多
- 矩阵没有高亮重点，变成信息平铺
- 页面看起来只是“表格换皮”

## 7. 成熟度模型 / 能力评估图

### 页面意图

用于表达：

- 企业攻防能力成熟度模型
- 主防体系建设阶段
- 运营能力等级评估
- 体系建设差距与提升方向

### 文本逻辑

至少应包含：

- 维度
- 等级 / 阶段
- 当前状态
- 建议动作或目标状态

### 图形绘画逻辑

- 可用分区树状、阶梯状、等级带状
- 不建议用普通 bullet list
- 若为多维成熟度，宜采用“维度块 + 等级定义”

### 呈现效果

- 要体现方法论感、评估感、模型感
- 页面应显得“有标准”，而不是“有观点”

### 文图协同

- 图形承担模型外形与等级关系
- 文本承担维度定义与等级说明
- 适合单页做重型方法论表达

### 推荐页型

- `18_domain_capability_map.svg`
- 次选：`12_grid.svg`, `16_table.svg`
- 兜底：深度改造后的 `03_content.svg`

### 高风险点

- 只有“能力点列表”，没有等级或维度结构
- 每个维度长度失衡，导致版面偏斜
- 没有当前状态和目标状态区分

## 8. 证据墙 / 背书页 2.0

### 页面意图

用于表达：

- 客户版图
- 赛事成绩
- 研究能力
- 资质认证
- 人才储备

### 文本逻辑

必须是“分类证据”，不是简单素材堆叠。

推荐语法：

```text
证据主题
→ 若干分组
→ 每组包含：名称 / 标签 / 关键结论
```

### 图形绘画逻辑

- 优先分组栅格、标签区块、逻辑墙面
- 不宜做成满页散落 logo
- 每组都需要有一个分组标题

### 呈现效果

- 重点是“厚重、全面、可信”
- 页面要有“证据陈列感”
- 不同类型证据之间要能分层，而不是所有元素同权

### 文图协同

- 图片 / logo / 证书图承担“证明”
- 文本承担“分类”和“结论”
- 每个组块最好只承载一个主题

### 推荐页型

- `12_grid.svg`
- 次选：`16_table.svg`, `07_data.svg`
- 兜底：深度改造后的 `03_content.svg`

### 高风险点

- 只是在堆 logo，没有结论
- 全页都同样密度，没有组块主次
- 分组标题缺失，导致阅读路径混乱

## 9. 多泳道实施链

### 页面意图

用于表达：

- 多角色并行推进的实施链
- 客户侧 / 长亭侧的协作推进
- 战前、战中、战后动作串联
- 方案落地过程中的阶段分工与交接

### 文本逻辑

它不是简单对比，而是“按阶段推进的双泳道或多泳道执行链”。

推荐语法：

```text
阶段 1：角色 A / 角色 B 的关键动作
→ 阶段 2：角色 A / 角色 B 的关键动作
→ 阶段 3：交接 / 收敛 / 结果确认
```

### 图形绘画逻辑

- 优先做横向阶段列 + 纵向泳道
- 同一阶段内的不同角色动作要上下对应
- 跨泳道交接应以短连接线或节点提示，不要做满页乱线

### 呈现效果

- 阅读应先按阶段横向推进，再看阶段内角色分工
- 页面要有推进感和“协同作战感”
- 重点阶段或关键交接点应突出

### 文图协同

- 图负责阶段顺序与角色分工
- 文负责每个阶段的关键动作说明
- 单页建议控制在 4 个阶段、2 条主泳道

### 推荐页型

- `09_comparison.svg`
- 次选：`05_case.svg`, `10_timeline.svg`
- 兜底：深度改造后的 `03_content.svg`

### 高风险点

- 只剩左右两列对照，没有“阶段推进”
- 每阶段动作过长，导致泳道卡片失衡
- 交接关系全靠长句说明，图示本身不成立

---

## 10. 分层攻击树 / 复杂结构图

### 页面意图

用于表达：

- 入口到目标的攻击树
- 复杂能力结构树
- 根因到控制点的分层拆解
- 一条主线下的多分支结构

### 文本逻辑

必须体现“层级”和“分支”，而不是普通模块列表。

推荐语法：

```text
根节点 / 主问题
→ 第一层分支
→ 第二层分支 / 关键节点
→ 目标层 / 影响层
```

### 图形绘画逻辑

- 优先用树状或层级分叉结构
- 顶部是根节点，中部是分支节点，底部是目标或结果节点
- 证据图或示意图应附在侧边，不要打断主树结构

### 呈现效果

- 要显得像“复杂结构被拆开了”，而不是“把 6 个点摆成两排”
- 主节点最强，分支次之，证据区再次之
- 阅读顺序必须一眼可追踪

### 文图协同

- 图负责层级与分支关系
- 文负责节点含义与结果解释
- 右侧可挂证据图或收束结论

### 推荐页型

- `08_product.svg`
- 次选：`17_service_overview.svg`, `18_domain_capability_map.svg`
- 兜底：深度改造后的 `03_content.svg`

### 高风险点

- 节点看起来只是卡片阵列，没有树感
- 根节点不突出，导致主线丢失
- 分支过多、连线过乱、右侧证据区与主树争夺焦点

---

## 11. 关键证据证明页 / 数据证明页

### 页面意图

用于表达：

- KPI + 证据图 + 结论分层
- 运营指标与攻击证据的联动说明
- 高密度但仍需稳态输出的数据证明页
- “数据不是为了展示，而是为了证明”

### 文本逻辑

至少应形成以下关系：

```text
指标锚点
→ 主证据图
→ 关键发现
→ 收束结论
```

### 图形绘画逻辑

- 顶部优先做 KPI 条带
- 中部保留主证据图 / 主图表区
- 侧边做分层解读卡，而不是把解释散落在图表周围

### 呈现效果

- 页面应显得像“证明中枢”，而不是“普通 KPI 页”
- 指标、主图、发现三层关系清楚
- 高密度但不能靠缩字号硬塞

### 文图协同

- KPI 负责建立判断锚点
- 主图负责承载证据
- 右侧结论卡负责解释与收束

### 推荐页型

- `07_data.svg`
- 次选：`19_result_leading_case.svg`, `12_grid.svg`
- 兜底：深度改造后的 `03_content.svg`

### 高风险点

- 只有 KPI 卡，没有主证明图
- 只有图表，没有结论卡
- 解读文案太长，压缩主图空间

---

## 12. 控制矩阵 / 治理看板

### 页面意图

用于表达：

- 域-状态-动作的治理矩阵
- 风险治理优先级看板
- 控制点盘点与收敛
- 重型治理结论页

### 文本逻辑

必须同时包含：

```text
治理域
→ 当前状态 / 风险等级
→ 关键治理动作
→ 验证或证据说明
```

### 图形绘画逻辑

- 左侧优先做纵向治理域矩阵
- 横向至少应区分状态、动作、证据三类信息
- 右侧要有结论收束，不要只剩表格

### 呈现效果

- 页面应显得像“治理看板”，不是“报表截图”
- 要有矩阵感、看板感、优先级感
- 重点域或高风险域应更突出

### 文图协同

- 矩阵负责盘点结构
- 右侧洞察区负责结论和优先级判断
- 单元格内尽量短标签，不写长段

### 推荐页型

- `16_table.svg`
- 次选：`12_grid.svg`, `07_data.svg`
- 兜底：深度改造后的 `03_content.svg`

### 高风险点

- 只做成普通表格，没有矩阵和看板感
- 右侧洞察太弱，导致整页像附件
- 一行信息过长，把矩阵变成文本墙

---

## 13. 证据挂载型复杂案例链

### 页面意图

用于表达：

- 节点上附带证据的复杂案例链
- 突破路径与关键证据并存的结果案例
- 攻击动作 + 旁证 + 结果证明的复合链路

### 文本逻辑

推荐语法：

```text
前提 / 挑战
→ 动作节点
→ 节点附证据
→ 结果证明
```

重点不是节点数量，而是“动作链”和“证据挂载”要同时成立。

### 图形绘画逻辑

- 主链应保持一条清晰推进线
- 证据块应附着在节点上方、下方或侧方
- 不要让证据块成为另一条主链

### 呈现效果

- 页面应有明显压迫感和调查感
- 关键突破节点与证据节点要形成组合焦点
- 最终结果要强于中间过程

### 文图协同

- 主链负责讲动作推进
- 证据块负责证明动作真实有效
- 结果区负责最后收束与价值放大

### 推荐页型

- `19_result_leading_case.svg`
- 次选：`05_case.svg`, `07_data.svg`
- 兜底：深度改造后的 `03_content.svg`

### 高风险点

- 节点有了，但证据仍然散落在页外
- 证据太多，主链被淹没
- 结果区不够强，页面变成过程流水账

---

## 四、复杂页的文图协同强度规则

### 4.1 协同等级

| 等级 | 定义 | 适用页型 |
|------|------|---------|
| 低耦合 | 图形偏装饰，文字可独立成文 | 不推荐用于高级正文页 |
| 中耦合 | 图负责结构，文负责解释 | 分层体系图、闭环图、证据墙 |
| 高耦合 | 图形节点与文案说明强绑定 | 攻击链、泳道图、矩阵图、实施流程图 |

### 4.2 模板默认策略

`security_service` 高级正文页默认采用：

- 大部分页面：中耦合
- 攻击链、复杂流程、矩阵图：局部高耦合

不推荐整套 deck 大量使用高耦合页面，否则阅读负担会过重。

---

## 五、复杂页的密度控制规则

高密度不等于堆满。复杂页要靠结构增密，而不是靠压缩字号增密。

### 5.1 通用控制规则

- 一级模块建议不超过 6 个
- 关键节点建议不超过 8 个
- 同页图形主结构不超过 2 套
- 同页图片证据不超过 6 张；攻击案例页可放宽到 8 张，但必须附着在节点附近
- 每个模块说明文字以 1-2 行为宜

### 5.2 该拆页的信号

出现以下任意一种情况，应优先拆页而非硬塞：

- 同页同时存在两套复杂主线
- 图形关系已经需要“解释图的图”
- 关键节点超过 8 个
- 单页既想讲体系，又想讲案例，又想讲数据
- 为了容纳内容不得不把正文字号压到 13px 以下

---

## 六、对 Strategist 的选页要求

Strategist 在使用 `security_service` 模板时，不应只判断“有没有合适页型”，还要判断“是否需要高级正文模式”。

先判断一层前提：

- 这页的复杂度是否来自文档内容本身，而不是来自主观追求“更高级”
- 若改成简单结构，是否会损失关键关系、关键证据或关键判断
- 若不会损失，就不应强行升级为高级正文页

### 6.1 触发高级页型的条件

若页面意图满足以下任一条件，应优先考虑高级正文页：

- 需要表达跨层关系
- 需要表达链路、闭环、映射、协同
- 需要同时承载 3 组以上模块，且它们不是简单并列
- 需要通过图示证明能力或实施机制
- 需要高密度但仍要求专业稳态

### 6.2 禁止的退化行为

- 明明是体系图，却退化成普通列表页
- 明明是攻击链，却退化成“背景 / 动作 / 结果”三卡片
- 明明是闭环机制，却画成线性流程
- 明明是证据墙，却只堆 logo 或截图
- 明明只是轻量说明页，却为了显得高级而硬套复杂图形

### 6.3 内容大纲输出字段要求

当使用 `security_service` 模板时，Strategist 不仅要选中模式，还要把选页依据写进项目级 `design_spec.md`。

除封面 / 目录 / 章节 / 结束页外，每个正文页至少输出：

- `页面意图`
- `证明目标`
- `高级正文模式`
- `优先页型`

若最终退到 `03_content.svg` 或 `11_list.svg`，还必须补一项：

- `回退原因`

也就是说，`03_content.svg` 与 `11_list.svg` 只能作为“解释过原因的例外页”，不能作为默认偷懒页。

### 6.4 命中信号优先于视觉便利

如果页面文本或结构已经明显出现以下信号，必须先判模式，再选页：

- `阶段 / 里程碑 / 建设路径 / 演进`
- `闭环 / 循环 / 持续优化 / 平战切换`
- `矩阵 / 映射 / 横纵维度`
- `体系 / 架构 / 平台 + 服务 + 结果`
- `客户版图 / 认证 / 赛事 / 研究成果 / 证据`
- `攻击树 / 根因拆解 / 分层结构`
- `多角色协同 / 阶段推进 / 实施链`
- `KPI + 主证据图 + 结论分层`
- `控制矩阵 / 治理看板 / 控制点`
- `节点证据 / 复杂案例链 / 结果证明`

不允许因为“普通内容页也能塞下”就跳过高级正文页。

---

## 七、对 Executor 的绘图要求

### 7.1 主结构优先

Executor 在画复杂页时，顺序必须是：

1. 先定页面主结构
2. 再放主节点 / 主模块
3. 再补连线 / 分组 / 视觉引导
4. 最后再填文本和证据图

禁止先把大量文字堆上去，再临时补线。

### 7.2 连线规则

- 连线只能承担关系提示，不承担主要阅读信息
- 连接线不可穿文字
- 分叉关系必须清晰，不要让连接线形成视觉迷宫

### 7.3 焦点规则

每个复杂页只能有 1 个主焦点，最多 2 个次焦点：

- 主焦点：标题下的核心图形骨架或结果 headline
- 次焦点：关键节点、结果指标、证据块

如果每个模块都一样重，页面就会显得乱。

---

## 八、复杂页 QA 补充项

除模板已有 QA 外，高级正文页还要额外检查：

- 图形类型是否与页面意图匹配
- 复杂图是否形成稳定阅读顺序
- 文本是否真正附着于图形结构，而非漂浮在外
- 连线、标签、截图是否存在局部打架
- 结果区是否弱于过程区
- 高密度是否由结构支撑，而不是靠缩字号解决

---

## 九、优先沉淀建议

如果后续要逐步升级模板，建议优先强化以下 10 类高级正文页：

1. 攻击链 / 案例链路图
2. 分层体系图
3. 闭环运营 / 机制图
4. 泳道协同图
5. 矩阵防御 / 映射图
6. 多泳道实施链
7. 分层攻击树 / 复杂结构图
8. 关键证据证明页 / 数据证明页
9. 控制矩阵 / 治理看板
10. 证据挂载型复杂案例链

这 10 类对 `security_service` 的专业度提升最明显，也最能拉开与普通方案 PPT 的差距。

---

## 十、结论

`security_service` 后续不应只理解为：

- “一套浅色长亭安服模板”

而应理解为：

- “一套能稳定产出安服能力证明型复杂正文页的模板系统”

因此，后续生成安服胶片时：

- 不只要判断页型
- 还要判断是否命中高级正文模式
- 不只要检查文案
- 还要检查图形逻辑、呈现效果、文图协同是否成立
