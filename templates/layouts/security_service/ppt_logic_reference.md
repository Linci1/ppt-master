## 一、模板定位

`security_service` 不是通用安服报告模板，而是 **长亭安服解决方案 / 能力证明型胶片模板**。

### 1.1 适用场景

- 长亭安服解决方案介绍
- 攻防演习 / HW 复盘与能力总结
- 安全运营体系介绍
- 客户案例证明型演示
- 对外能力背书型胶片

### 1.2 不适合的场景

- 纯内部周报 / 月报
- 普通项目日报
- 以长段叙述为主的研究汇报
- 无明显品牌表达需求的通用 PPT

## 二、历史素材的真实章节节奏

历史素材不是普通“背景 → 方案 → 实施 → 总结”的五段式，而是更强的品牌化章节节奏。

### 2.1 推荐章节骨架

```text
01 品牌与理念
02 安服体系总览
03 “攻”能力
04 “防”能力
05 “培”能力
06 安全运营能力
07 案例证明
08 优势与能力背书
09 结束页
```

### 2.2 章节组织原则

- **先建立认知**：先讲长亭是谁、安服方法论是什么
- **再讲体系**：说明整套安服体系如何运转，而不是直接进入零散能力点
- **再讲分域能力**：按 `攻 / 防 / 培 / 运营` 逐步展开
- **再给案例证明**：用结果导向案例证明能力不是抽象概念
- **最后做背书收束**：客户覆盖、赛事成绩、研究能力、资质认证共同建立可信度

### 2.3 生成时的默认顺序

当源内容属于安服 / HW / 安全运营类主题，且用户未要求特殊结构时，优先按以下顺序组织内容：

1. 品牌理念或业务定位
2. 服务体系总览
3. 核心域能力展开
4. 结果导向案例
5. 能力与资质背书
6. 收束页

## 三、页面意图定义

### 3.1 各类页面不是“装内容”，而是完成特定表达任务

| 页面类型 | 表达任务 | 典型页型 |
|---------|----------|---------|
| 品牌理念页 | 建立“我们是谁、方法论是什么” | `17_service_overview.svg`, `12_grid.svg` |
| 体系总览页 | 建立“平台 + 服务 + 结果”的整体理解 | `17_service_overview.svg` |
| 分域能力页 | 解释 `攻 / 防 / 培 / 运营` 各自能力边界和价值 | `18_domain_capability_map.svg`, `06_tactics.svg` |
| 方法论 / 闭环页 | 表达流程、平战切换、运营闭环 | `18_domain_capability_map.svg`, `10_timeline.svg` |
| 案例页 | 证明能力有效，突出结果与价值 | `19_result_leading_case.svg`, `05_case.svg` |
| 数据页 | 用指标强化结论 | `07_data.svg`, `13_highlight.svg`, `16_table.svg` |
| 背书页 | 建立可信度，形成证据墙 | `12_grid.svg`, `16_table.svg`, `07_data.svg` |

### 3.2 页面表达原则

- 每页都应回答“这页要证明什么”
- 如果一页只是在“介绍”，大概率不够像历史胶片
- 如果一页没有明确结论、结构或证据支撑，应优先重组而不是直接排版

## 四、标题逻辑

### 4.1 标题必须带表达立场

历史素材中的标题往往不是中性描述，而是带有明显价值判断或章节识别。

### 4.2 推荐标题模式

| 类型 | 模式 | 示例 |
|------|------|------|
| 分域能力页 | `“X”——结论/体系名` | `“攻”——自动化攻防体系` |
| 运营页 | `【主题】——价值表达` | `【安全运营】攻防螺旋下的“左右互搏”` |
| 案例页 | `案例X：客户/行业——结果结论` | `案例1：某省电科院——自动化工具支撑季度演练持续降本增效` |
| 背书页 | 结论式名词短语 | `攻防赛事大满贯`、`行业领先的完整资质认证` |
| 理念页 | 方法论 / 体系定位 | `长亭运营与服务基本理念`、`安全服务体系` |

### 4.3 禁止的标题倾向

- `案例介绍`
- `项目背景`
- `解决方案说明`
- `能力展示`
- `客户案例`

这些标题信息量太低，容易把胶片做成通用汇报。

## 五、正文写法

### 5.1 基本原则

- **结论先行**：先讲结果，再讲原因/动作
- **短句优先**：优先标签、短句、数据点
- **证据表达**：尽量给出量化结果、场景词、动作词
- **避免长段落**：不要连续三行以上自然段说明

### 5.2 句式建议

适合本模板的正文更像：

```text
重点场景：面向国家级攻防演习与常态化蓝军建设
核心能力：自动化发现、攻击知识沉淀、实战带训
结果价值：持续降本增效、稳定产出重大成果
```

不适合本模板的正文：

```text
我们通过多年的发展积累了丰富的经验，在长期项目实践中逐步形成了体系化的方法论，
能够根据客户需求提供较为全面的服务支持。
```

### 5.3 案例页写法

案例页建议固定为三段式之一：

- `背景 / 动作 / 结果`
- `挑战 / 方案 / 成效`
- `现状 / 提升举措 / 指标变化`

同时要求：

- 页头标题必须带结果导向
- 页面内至少有一处可量化结论
- 不要把案例页写成流水账过程页

## 六、绘图逻辑

### 6.1 历史素材的核心绘图套路

| 内容意图 | 推荐图示 | 说明 |
|----------|----------|------|
| 体系总览 | 分层结构图 / 平台 + 服务图 | 强调能力全景 |
| 分域能力 | 能力地图 / 模块图 | 强调场景、模块、价值之间的映射 |
| 运营闭环 | 环形闭环 / 链路图 | 强调发现、验证、处置、优化 |
| 案例证明 | 三段证明链 | 强调挑战、动作、结果 |
| 背书 | 证据墙 / 矩阵 / 荣誉墙 | 强化可信度 |

### 6.2 推荐的视觉构成

- 蓝色负责结构骨架
- 橙色负责结果、强调、结论
- 浅蓝和浅灰做辅助分层
- 卡片、连线、分组、标签共同承担信息组织，不依赖纯文字说明

### 6.3 不推荐的退化形式

- 能力页退化成普通 bullet list
- 运营页退化成四个孤立卡片，没有闭环感
- 案例页只剩 2x2 文本块，没有结果主张
- 背书页写成表格或长列表，没有视觉证据组织

## 七、页型选择优先级

### 7.1 高优先级页型

| 页型 | 适用内容 |
|------|----------|
| `17_service_overview.svg` | 品牌理念、体系总览、平台与服务关系 |
| `18_domain_capability_map.svg` | `攻 / 防 / 培 / 运营` 能力展开、流程闭环 |
| `19_result_leading_case.svg` | 结果导向案例页 |

### 7.2 中优先级页型

| 页型 | 适用内容 |
|------|----------|
| `07_data.svg` | 多指标 + 图表解释 |
| `13_highlight.svg` | 单一核心 KPI |
| `16_table.svg` | 详细数据与关键发现 |
| `06_tactics.svg` | 方法论 / 战法 / 重点场景 |
| `12_grid.svg` | 背书矩阵 / 多维能力卡片 |

### 7.3 低优先级兜底页型

| 页型 | 使用条件 |
|------|----------|
| `03_content.svg` | 无合适结构页时才使用 |

| `09_comparison.svg` | 仅在确实需要强对比时使用 |
| `14_fullimage.svg` | 仅在需要视觉冲击时使用 |

## 八、每类章节的推荐打法

### 8.1 品牌与理念

- 用一页解释安服理念或业务定位
- 用一页解释服务体系全景
- 少做抽象口号，多做结构表达

### 8.2 `攻 / 防 / 培 / 运营`

每个章节最好至少包含：

1. 一页能力概览
2. 一页方法论/场景页
3. 一页案例或数据支撑

### 8.3 案例证明

案例章节应优先覆盖：

- 客户场景
- 我方动作
- 成果指标
- 对客户价值

### 8.4 背书收束

背书页建议优先从以下维度中选 2-4 类：

- 地域与专家团队覆盖
- 客户版图
- 攻防赛事成绩
- 安全研究成果
- 云安全服务能力
- 资质认证
- 人才建设与储备

## 九、案例级复杂正文补充要求

当页面命中复杂页时，除已有页型、图形和证据规则外，还必须额外满足：

- 标题、主图、证据说明、页尾判断必须共享同一条 `论证主线`
- 每个主证据都必须能映射到某个节点、关系或结果，不能漂浮在页面边缘充当装饰
- 同一页只能承担一个主复杂任务：解释结构 / 解释链路 / 解释机制 / 证明结果 / 收束治理
- 文案必须完成从“原文缩写”到“顾问判断”的压缩，避免说明式、介绍式、拼接式表达
- 相邻复杂页之间必须存在推进关系，避免两页都在重复证明同一件事

这些要求的详细做法，统一以 `sample_grade_content_system.md` 为补充执行基线。

## 十、Anti-Patterns

以下情况即使排版工整，也不符合模板目标：

1. 全文像通用汇报，而不像长亭安服胶片
2. 标题没有结论感，只有中性主题词
3. 重要能力页没有图示结构，只是简单列表
4. 案例页没有结果，只描述过程
5. 背书页没有“证据墙”结构，只是信息堆砌
6. 一页内容过多，导致文字贴边、断句难看、信息密度失衡

---

## 十一、生成时的执行提示

当使用 `security_service` 模板时，Strategist 和 Executor 阶段都应默认执行以下动作：

- 先判断内容属于哪个章节意图，而不是先找一个能塞下内容的页型
- 先确定页面要“证明什么”，再决定布局
- 优先调用高优先级页型
- 每页生成后检查：
  - 标题是否结论先行
  - 图示是否贴合历史素材表达
  - 文本是否拥挤、贴边、越界
  - 页码 / 底部装饰 / logo 是否冲突
