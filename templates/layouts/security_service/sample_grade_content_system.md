# 长亭安服案例级复杂正文系统

> 用途：把 `security_service` 从“已有复杂页骨架”继续推进到“更接近历史案例胶片的复杂正文表达系统”。
>
> 核心目标不是把页面做得更花，而是让复杂页在 **论证语言、证据锚点、图形组织、跨页推进** 四个层面都更像成熟安服胶片。

---

## 一、这份文档解决什么问题

当前复杂页能力已经具备：

- 能识别攻击链、矩阵、闭环、证据墙等高级正文模式
- 能生成比普通卡片页更复杂的 SVG 结构
- 能做基础的文本、布局和模板 QA

但和历史案例素材相比，仍常见以下差距：

- 图做复杂了，但主判断不够像安服顾问在做推演
- 证据放进来了，但仍像“配图”而不是“证明节点”
- 节点、标题、页尾判断之间还没有形成一条完整论证线
- 单页看起来不差，但跨页之间没有形成“概览 → 推进 → 证明 → 收束”的强节奏

因此，`security_service` 后续生成复杂正文页时，除已有模板规则外，还必须额外遵循本文件。

## 二、案例级复杂正文的四层系统

### 2.1 论证语言层

每个复杂页都必须把原文事实压缩成一条清晰论证线，而不是只把事实分块摆上去。

推荐总公式：

```text
现象 / 入口
→ 放大机制 / 关键条件
→ 结果 / 影响
→ 管理判断 / 动作要求
```

复杂页的标题、节点、证据说明、页尾总结，都必须服务于同一条论证线。

### 2.2 证据锚点层

案例素材中的好页面，不是“有几张截图”，而是“每张截图都有锚点”。

每个证据都必须回答：

- 它在证明哪个节点？
- 它在证明哪个关系？
- 它在证明哪个结果？

如果一张图回答不了这三个问题中的任何一个，它就不应进入复杂页主结构。

### 2.3 图形组织层

复杂图不是为了密，而是为了把一种复杂认知压缩得更快读懂。

推荐原则：

- 图负责结构，不负责解释全部细节
- 文负责判断、机制和收束
- 主图必须先让读者一眼看到主线，再让读者第二眼看细节
- 同一页只允许一个主复杂任务，不允许一页同时承担两套互不相干的复杂认知

### 2.4 跨页推进层

案例级复杂正文的专业感，很大一部分来自相邻页面之间的推进关系。

典型强节奏：

```text
概览页：先定义问题结构
→ 推进页：解释链路 / 机制 / 分层
→ 证明页：拿出关键证据或结果
→ 收束页：给出治理动作 / 优先级 / 闭环要求
```

如果连续两页都只是在“重复证明同一件事”，整套胶片就会显得松。

## 三、从原文中提取复杂页的规则

### 3.1 先看是否存在“结构性内容”

只有原文存在以下信号时，才建议进入复杂页：

- 多入口、多路径、多节点最终汇聚到同一结果
- 明确的放大条件、依赖条件、权限变化
- 多角色、多阶段、多域协同
- 风险域与控制动作之间存在映射关系
- 既有数据结论，又有原始证据截图或案例结果图

如果原文只是单一结论、轻量建议、普通概览，不应强做复杂页。

### 3.2 再判断这页的主复杂任务

每个复杂页只能承担一个主任务：

- 解释结构
- 解释链路
- 解释机制
- 证明结果
- 收束治理

不能一页同时“讲体系 + 讲案例 + 讲数据 + 讲建议”。

### 3.3 最后再选结构

不是“先看到一个复杂模板就套进去”，而是：

```text
先抽主判断
→ 再抽分判断
→ 再抽证据锚点
→ 最后决定用链路 / 分层 / 矩阵 / 闭环 / 泳道
```

## 四、六类高价值复杂页的案例级打法

### 4.1 风险结构总览页

**页面任务**

- 把离散问题压缩成可理解的风险结构

**必须回答**

- 风险不是零散点，而是分布在哪些域
- 哪些域最可能向同一结果收敛

**推荐论证线**

```text
问题分布
→ 高风险域
→ 汇聚方向
→ 管理层应关注的结构性判断
```

**图形打法**

- 分层结构图
- 中心汇聚图
- 风险域地图

**文案要求**

- 标题必须是结构判断，而不是“风险概览”
- 模块标题要写成域判断，如“管理面暴露构成横向放大器”

**禁止**

- 普通 6 条 bullet
- 只按漏洞类型机械分类

### 4.2 攻击链 / 路径推演页

**页面任务**

- 证明一条突破链确实成立，并解释为什么能走深

**必须回答**

- 从哪里进
- 中间如何放大
- 最终打到哪里
- 为什么这不是单点问题

**推荐论证线**

```text
入口
→ 动作
→ 放大条件
→ 控制结果
→ 管理判断
```

**图形打法**

- 单主链
- 多入口汇聚主链
- 主链 + 侧挂证据

**文案要求**

- 节点写“对象 + 动作 / 状态变化”
- 页尾必须有一句“这条链路说明什么”

**禁止**

- 写成操作流水账
- 只有步骤，没有机制

### 4.3 证据证明页

**页面任务**

- 用有限证据，证明前面那条判断不是抽象推演

**必须回答**

- 哪个证据最关键
- 它分别证明哪个节点 / 结果
- 证据与标题之间是什么关系

**推荐论证线**

```text
前页判断
→ 本页主证据
→ 关键结果
→ 管理层可接受的结论
```

**图形打法**

- 主证据区 + 结果区 + 说明区
- 证据挂在结果节点附近，而不是散落

**文案要求**

- 每张证据图旁必须有一句证明句
- 主证据不超过 3 个

**禁止**

- 截图拼贴墙
- 证据说明只写“如下图所示”

### 4.4 控制矩阵 / 治理看板页

**页面任务**

- 把问题翻译成治理优先级和控制动作

**必须回答**

- 哪些域要先做
- 每个域做什么
- 如何验收
- 为什么这样排优先级

**推荐论证线**

```text
主风险域
→ 关键控制动作
→ 验证方式
→ 闭环标准
```

**图形打法**

- 风险域 × 控制动作矩阵
- 优先级看板
- 分层治理表

**文案要求**

- 每个动作都要可验证
- 完成标准必须是复测 / 复核 / 留痕后的状态，而不是口号

**禁止**

- 任务清单式表格
- 只有动作，没有闭环标准

### 4.5 运营闭环 / 服务机制页

**页面任务**

- 说明服务不是静态动作，而是持续运行机制

**必须回答**

- 机制如何循环
- 每一环如何接上一环
- 长亭在其中承担什么专业动作

**推荐论证线**

```text
发现
→ 验证
→ 处置
→ 优化
→ 下一轮输入
```

**图形打法**

- 闭环
- 环形流程
- 平战结合双环

**文案要求**

- 不写成普通流程图
- 各环节要写清产出，而不是只写动作名

**禁止**

- 四个静态卡片冒充闭环
- 各步骤互不咬合

### 4.6 结果导向案例页

**页面任务**

- 把案例写成“挑战 - 动作 - 结果 - 价值”的证明链

**必须回答**

- 场景是什么
- 我方做了什么关键动作
- 带来了什么结果
- 这个结果说明我方具备什么能力

**推荐论证线**

```text
客户场景
→ 长亭动作
→ 结果指标 / 结果事实
→ 能力证明 / 客户价值
```

**图形打法**

- 结果 headline + 三段证明链
- 案例链 + 指标卡 + 价值总结

**文案要求**

- 标题必须结果导向
- 页尾判断要回到“能力已被证明”

**禁止**

- 过程流水账
- 没有指标、没有结果、没有价值归因

## 五、案例级文本压缩规则

### 5.1 标题压缩

标题优先写：

- 结构判断
- 结果判断
- 管理判断

避免：

- 场景描述
- 中性栏目名
- 大而空的口号

### 5.2 节点压缩

节点文案优先写：

- `对象 + 动作`
- `对象 + 状态变化`
- `条件 + 风险作用`

例如：

- `管理面暴露`
- `后台访问成立`
- `云凭证泄露`
- `终端控制落点`

### 5.3 证据说明压缩

证据说明统一回答：

- 该图证明什么
- 为什么它重要

推荐句型：

- `该图证明后台入口已可被稳定访问`
- `该截图证明数据库权限已触达并可继续放大利用`

### 5.4 页尾收束压缩

页尾不复述页面内容，而要给管理判断：

- 这页最终说明了什么
- 为什么值得优先处理
- 下一页将推进什么

## 六、案例级复杂页的跨页组合

### 6.1 攻防演练 / 渗透总结

```text
风险结构总览
→ 关键攻击路径页
→ 证据证明页
→ 控制矩阵 / 整改闭环页
```

### 6.2 安服能力胶片

```text
体系总览
→ 分域能力机制页
→ 结果导向案例页
→ 证据墙 / 背书页
```

### 6.3 运营与治理专题

```text
问题结构页
→ 放大机制页
→ 运营闭环页
→ 优先级治理页
```

## 七、强制反模式

出现以下任意一类情况，都说明还没有达到案例级复杂正文：

1. 复杂图已经画出来，但标题、节点、页尾判断不是同一条主线
2. 证据存在，但没有节点锚点或证明句
3. 页面看起来很满，但读者第一眼抓不到主判断
4. 一页同时承担多个复杂任务，导致结构互相抢焦点
5. 相邻两页都在重复证明同一件事，没有推进关系
6. 文案仍像原报告压缩版，而不像安服专家判断

## 八、落地要求

命中 `security_service` 且使用复杂页时，至少要把本文件落实到以下位置：

- `design_spec.md`：决定该页是否值得进入复杂模式、它在整套 deck 中承担什么角色
- `notes/complex_page_models.md`：补齐论证主线、证据分级、文本压缩计划
- `text_prompt_snippets.md`：约束具体生成时的文案与论证语言
- `generation_checklist.md` / `qa_profile.md`：在导出前做强制检查
- `soft_content_qa_framework.md`：专门拦截“有其表、无其实”的软性内容问题
- `soft_content_rewrite_strategies.md`：软问题命中后的标题、模块、证据、页尾重写动作

只有当这些层都打通，复杂页才会真正接近历史案例胶片的专业程度。
