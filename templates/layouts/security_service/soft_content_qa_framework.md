# 长亭安服软性内容 QA 框架

> 用途：专门拦截“初看没问题，但细看表述逻辑混乱”的软性问题。
>
> 这份框架不替代布局 QA，而是补上安服正文页最容易出现的“有其表、无其实”的内容风险。

---

## 一、这份框架要解决什么

有些页面在版式、对齐、Logo、安全区上都没有明显问题，但仍会让人觉得“不够像案例素材”。

通常不是因为页面不复杂，而是因为：

- 主判断不够硬
- 模块之间不是一条逻辑线
- 证据存在但没有证明作用
- 图形看起来复杂，但只是信息整理
- 页尾没有真正完成管理收束

因此，`security_service` 在复杂页和高信息密度正文页中，必须额外做软性内容 QA。

## 二、八类高频软问题

### 2.1 主线漂移

表现：

- 标题在讲 A，模块在讲 B，页尾却落到 C
- 页面里能看到多个中心结论互相抢焦点

判定问题：

- 读者读完后，不知道这页到底要证明什么

修正方向：

- 强制收敛为 1 个主判断
- 删掉不服务主判断的模块

### 2.2 模块关系混乱

表现：

- 上下模块只是并列罗列，没有因果 / 递进 / 汇聚关系
- 讲的时候要靠口头解释，页面自己说不清

判定问题：

- 页面像整理页，而不是论证页

修正方向：

- 先明确模块关系类型：并列 / 因果 / 放大 / 汇聚 / 闭环
- 若关系说不清，回退为更简单结构

### 2.3 标题、模块标题、页尾判断不统一

表现：

- 标题是结论句，但模块标题是栏目名
- 模块讲的是过程，页尾写的是泛泛而谈的建议

判定问题：

- 页面内部语言级别不统一

修正方向：

- 标题写主判断
- 模块标题写分判断
- 页尾写管理判断 / 动作要求

### 2.4 证据悬空

表现：

- 页面有截图、有数据，但说不清这些图分别证明什么
- 证据放在边上，只起到“显得丰富”的作用

判定问题：

- 页面看起来像证明页，实际还是装饰页

修正方向：

- 每个证据都要对应节点 / 关系 / 结果
- 每个证据旁都补一句证明句

### 2.5 图文错位

表现：

- 图在讲结构，文在讲另一个层面的信息
- 文字只是重复图中节点名，没有解释价值

判定问题：

- 图和文同时存在，但没有互相增强

修正方向：

- 图只负责结构
- 文只负责判断、机制、收束
- 删掉重复节点名的解释句

### 2.6 结果与动作断裂

表现：

- 页面写了很多问题和动作，但没讲为什么这些动作能解决问题
- 页尾建议像任务清单，不像治理判断

判定问题：

- 页面只能解释“发现了什么”，不能解释“应该怎么办”

修正方向：

- 每个动作都要对应某个风险域或结果面
- 写清验证方式和完成标准

### 2.7 跨页推进断裂

表现：

- 这一页和下一页都在重复证明同一件事
- 页尾没有给下一页留承接点

判定问题：

- 单页也许成立，但整套 deck 没有节奏

修正方向：

- 给复杂页标注页面角色
- 页尾判断要么收束，要么为下一页推进铺路

### 2.8 复杂外形掩盖了整理本质

表现：

- 节点很多、箭头很多、颜色很多
- 但本质还是把原文拆块摆开

判定问题：

- 页面属于“复杂排版”，不是“复杂知识表达”

修正方向：

- 回答“为什么这页必须复杂”
- 如果复杂结构没有明显提升理解效率，就回退页型

## 三、软性内容 QA 的标准问法

每页至少要回答以下 6 个问题：

1. 这页唯一主判断是什么？
2. 模块之间是什么关系？
3. 标题、模块标题、页尾判断是不是同一条线？
4. 每个证据分别在证明什么？
5. 图和文是否在互相增强，而不是互相重复？
6. 这页读完后，管理层应得到什么判断或动作要求？

只要其中任一问答不上来，这页就不能视为通过。

---

## 四、适用场景

以下页面默认应使用本框架：

- 问题分析页
- 攻击链页
- 证据证明页
- 治理矩阵页
- 结果导向案例页
- 高信息密度的能力证明页

## 五、落地位置

这份框架应同时落到：

- `text_prompt_snippets.md`：让生成前先做软性自问
- `generation_checklist.md`：让生成阶段强制检查软问题
- `qa_profile.md`：让导出前把软问题视为不通过项
- `complex_page_logic_qa_checklist.md`：让复杂页额外接受软逻辑审查
- `soft_content_rewrite_strategies.md`：命中软问题后的页面级重写动作
