# 长亭安服软性问题重写策略

> 用途：当页面已经通过基础排版 QA，但仍存在“有其表、无其实”的软性问题时，提供一套明确的改写动作。
>
> 目标：不是只指出问题，而是把问题直接翻译成可执行的重写步骤，尤其作用于标题、模块标题、证据说明、页尾判断和模块关系。

---

## 一、使用时机

当页面出现以下任一情况时，不应只做微调，而应进入重写策略：

- 主判断不清
- 标题、模块标题、页尾判断不一致
- 模块关系说不清
- 证据存在但无法证明主判断
- 图文咬合不深
- 页尾没有形成管理收束
- 页面只能靠口头解释成立

---

## 二、通用重写顺序

遇到软性问题时，默认按以下顺序重写，而不是直接改文案细枝末节：

```text
先收敛主判断
→ 再重排模块关系
→ 再重写标题与模块标题
→ 再重绑证据
→ 最后重写页尾收束
```

如果第一步都做不到，后面的微修没有意义。

---

## 三、八类软问题的重写动作

### 3.1 主线漂移

**典型信号**

- 标题像风险判断
- 中间模块像过程说明
- 页尾像整改建议
- 三者各讲各的

**重写动作**

1. 先写一句唯一主判断
2. 删除不直接支撑主判断的模块
3. 把剩余模块改写为 2-3 个分判断
4. 用分判断重新命名模块标题
5. 页尾改写为对主判断的管理收束

**推荐改法**

- 标题：写主判断
- 模块标题：写分判断
- 页尾：写“这页最终说明什么 / 为什么该优先处理”

### 3.2 模块关系混乱

**典型信号**

- 模块都是名词块
- 看不出先后、因果、汇聚还是闭环

**重写动作**

1. 先强制判定模块关系类型：并列 / 因果 / 放大 / 汇聚 / 闭环
2. 若关系不止一种，只保留最核心的一种
3. 根据关系类型重排模块顺序与视觉主次
4. 模块标题中加入关系意味，而不是纯栏目名

**推荐改法**

- 因果页：按“原因 → 放大 → 结果”排
- 汇聚页：按“多问题 → 共享结果”排
- 闭环页：按“发现 → 验证 → 处置 → 复测”排

### 3.3 标题、模块标题、页尾判断不统一

**典型信号**

- 标题有结论感
- 模块标题是“问题一 / 关键点 / 情况说明”
- 页尾是泛泛建议

**重写动作**

1. 标题保留主判断层级
2. 模块标题全部改成“小结论”或“机制判断”
3. 页尾统一改成管理判断或动作要求
4. 删除所有只是栏目导航作用的空标题

**推荐改法**

- 不写：`情况说明` / `问题一` / `风险点`
- 改写为：`管理面暴露放大了横向成本` / `口令复用让入口可持续放大`

### 3.4 证据悬空

**典型信号**

- 有截图、有日志、有图表
- 但说不清每个证据在证明什么

**重写动作**

1. 给每个证据指定挂点：节点 / 关系 / 结果 / 分判断
2. 删除无法挂点的证据
3. 每个证据旁补一句证明句
4. 主证据优先贴近结果节点和关键动作节点

**推荐改法**

- 证据说明统一回答：`该图证明什么`
- 不写：`如下图所示`
- 改写为：`该图证明后台入口已可被稳定访问`

### 3.5 图文错位

**典型信号**

- 图在讲结构
- 文在讲背景或另一层解释
- 两边没有互相证明

**重写动作**

1. 先定义图负责什么：结构 / 链路 / 分层 / 闭环
2. 再定义文负责什么：判断 / 机制 / 收束
3. 删除所有重复图中节点名的文字
4. 把文字改成“为什么这结构成立”或“这结构意味着什么”

**推荐改法**

- 图旁文字尽量写机制句或判断句
- 不重复节点名

### 3.6 结果与动作断裂

**典型信号**

- 页面前半讲问题
- 页尾突然给建议
- 中间缺少“为什么这些动作正好对这组问题有效”的桥梁

**重写动作**

1. 先按风险域或结果面分组动作
2. 每个动作都写清对应对象
3. 补充验证方式和完成标准
4. 页尾收束改写为优先级判断，而不是任务清单

**推荐改法**

- 写：`先收口入口面，再同步整改管理面与凭证条件`
- 不写：`建议加强管理、提升意识、完善制度`

### 3.7 跨页推进断裂

**典型信号**

- 上一页和下一页都在重复证明一件事
- 这一页结尾没有给下一页留推进钩子

**重写动作**

1. 给当前页重新确认页面角色：概览 / 推进 / 证明 / 收束
2. 根据页面角色重写页尾判断
3. 若当前页已经完成证明，则下一页改去讲机制或治理
4. 若当前页只建立结构，则页尾明确写“下一页将证明什么”

### 3.8 复杂外形掩盖整理本质

**典型信号**

- 结构很多
- 但本质是原文分块搬运

**重写动作**

1. 先回答“为什么这页必须复杂”
2. 若答不上来，降级页型
3. 若答得上来，删掉不能提升理解效率的结构噪音
4. 保留一条最核心阅读主线

**推荐改法**

- 复杂页不追求模块多，而追求主线更快可读

## 四、五个关键部位的直接改写规则

### 4.1 标题改写

标题必须优先写：

- 主判断
- 结果判断
- 管理判断

避免：

- 中性主题词
- 流水账标题
- 章节感过强但判断感过弱的标题

### 4.2 模块标题改写

模块标题必须优先写：

- 分判断
- 机制判断
- 风险作用

避免：

- `模块一`
- `问题域`
- `情况说明`

### 4.3 证据说明改写

证据说明统一改成：

- 该图证明什么
- 它支撑哪个节点 / 结果 / 分判断

### 4.4 页尾收束改写

页尾必须承担以下其中之一：

- 管理判断
- 风险判断
- 动作要求
- 下一页承接

避免：

- 重复页面内容
- 空泛建议
- 只做总结，不做判断

### 4.5 模块顺序改写

优先按以下顺序重排：

- 风险页：现象 → 机制 → 结果
- 证据页：判断 → 证据 → 收束
- 治理页：风险域 → 动作 → 验证 → 闭环
- 案例页：场景 → 动作 → 结果 → 证明点

---

## 五、重写完成后的复核问法

页面重写后，必须重新回答以下问题：

1. 标题是不是这页唯一主判断？
2. 模块标题是不是都变成了分判断或机制判断？
3. 每个证据是不是都能说出它证明什么？
4. 页尾是不是给了管理判断或动作要求？
5. 如果去掉讲解，页面自己还能不能成立？

若仍有任一项答不上来，说明这页还没有真正修完。
